# Spotlight 全行跟踪修复与重复声明清理

## 背景

本次任务承接上一会话（首页首屏设计工作）的遗留问题，涉及两个独立 bug：

1. **`ParamOverrideEditorModal.jsx` 构建报错**：上一会话末尾发现 `SYNC_TARGET_TYPE_OPTIONS` 已被修复，但 `LEGACY_TEMPLATE` 和 `OPERATION_TEMPLATE` 仍同时存在 import 和本地 `const` 声明两份，导致 dev server 启动时构建失败。

2. **Spotlight 光晕只在第一行右侧三分之一范围内跟随鼠标**：用户反映首页 Hero 区域的鼠标跟踪光晕无法覆盖第一行左侧（文案区域），只在机器人（Spline 3D canvas）所在的右列范围内有效。

## 实际改动

### 1. `ParamOverrideEditorModal.jsx` 重复声明清理

删除了本地重复的两个 `const` 定义：

- `LEGACY_TEMPLATE`（同时被从 `paramOverrideEditorModalConstants.js` import）
- `OPERATION_TEMPLATE`（同时被从 `paramOverrideEditorModalConstants.js` import）

这两个常量在上一会话的大文件拆分中被提取到 constants 文件，但主文件里的旧定义未同步删除，产生双声明报错。

### 2. `spotlight.jsx` 事件监听重构

**第一步**：将 `parentElement.addEventListener('mousemove')` 改为 `window.addEventListener('mousemove')`，解决依赖事件冒泡的不稳定性。

**第二步**：添加 `if (!event.isTrusted) return;` 过滤合成事件。

**第三步**：移除 rAF 节流层，直接调用 `mouseX.set()` / `mouseY.set()`，由 framer-motion 自身的调度器管理帧合并。

最终的 `handleMouseMove` 结构：

```js
const handleMouseMove = useCallback((event) => {
  if (!event.isTrusted || !parentElement) return;
  const rect = parentElement.getBoundingClientRect();
  const inBounds = event.clientX >= rect.left && event.clientX <= rect.right &&
                   event.clientY >= rect.top && event.clientY <= rect.bottom;
  setIsHovered(inBounds);
  if (!inBounds) return;
  mouseX.set(event.clientX - rect.left);
  mouseY.set(event.clientY - rect.top);
}, [mouseX, mouseY, parentElement]);
```

## 根本原因定位过程

排查分三步进行：

**第一步**：上一会话已将 `<Spotlight>` 从 `ald-home-hero__primary-background`（position: absolute 子层）提升为 `<section>` 的直接子级，修复了 Spotlight 的 `useEffect` 读取内联样式（空字符串为 falsy）后强写 `position: relative`、覆盖背景层绝对定位导致布局错乱的问题。但光晕仍只在右列有效。

**第二步**：将监听改到 `window` 层。Playwright 截图确认：在 x=300（左列）时 spotlight `is-visible` 已被设置，但位置仍然错误（停留在前一帧坐标或初始位置）。

**第三步**：通过 Playwright 逐步定位，发现位置被覆盖——每次真实 `mousemove`（如 x=300）之后，spotlight 最终停到机器人 canvas 区域的 x 值。

**根本原因**：`spline-scene.jsx` 监听 `window` 上的 `pointermove`，将真实鼠标坐标夹取（clamp）到 Spline canvas 范围内后，合成一个 `new MouseEvent('mousemove', { bubbles: true, clientX: 夹取后坐标 })` 派发给 canvas。这个合成事件 `bubbles: true`，会从 canvas 冒泡至 `window`，被 spotlight 的 `window.addEventListener('mousemove')` 接到，用夹取后的 x 坐标（≈ robot canvas 左边界）覆盖真实鼠标位置。效果：无论真实鼠标在哪里，spotlight 最终都被拉回到机器人 canvas 区域。

**修复**：`event.isTrusted === false` 代表合成事件（`dispatchEvent()` 派发的），返回即可。

## 验证

Playwright 截图（port 5173）确认三个位置正确跟踪：
- x=300 → 光晕在文案左列
- x=700 → 光晕在中间
- x=1100 → 光晕在机器人右列

构建验证：`bun run build` 通过，无 duplicate declaration 错误。

## 关键判断与边界

1. `isTrusted` 是 Web 标准属性，`true` = 用户真实输入，`false` = JS `dispatchEvent()` 合成；这是过滤合成事件的标准做法，不依赖私有 API。
2. 移除 rAF 节流不会导致性能问题——framer-motion 的 `MotionValue.set()` 内部已有帧合并机制，不需要外层再做 rAF 节流。
3. 本次没有修改 `spline-scene.jsx` 的合成事件逻辑（该逻辑是为了让 Spline 的头部跟随交互正常工作），而是在 spotlight 侧过滤，避免对 Spline 交互产生副作用。
