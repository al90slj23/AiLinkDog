# gogogo 本地开发自动重载接入

## 背景

当前仓库已经把 `./gogogo.sh dev` 作为本地开发主入口，并通过 `tmux` 同时拉起前后端开发环境。

在这次调整前：

- 前端已经使用 `vite`，大多数页面与样式改动可以通过 HMR 或自动刷新直接反馈
- 后端开发态仍然是直接启动 Go 进程，修改 `.go` 文件后需要手动停掉再重启，反馈链路明显慢于前端

本次工作的目标不是引入运行时热替换，而是把后端本就必须发生的“重新编译 + 重启”收敛到统一开发入口下，并让 `gogogo` 脚本在多 worktree 场景里仍然可用。

## 本次事实变更

本次已经落地并验证的事实如下：

1. 后端本地开发启动方式从 `go run main.go` 切换为 `air`
2. 前端本地开发方式保持不变，仍然是 `bun install && bun run dev`
3. `./gogogo.sh stop` 与 `gogogo.1.sh` 中的重启路径，已经接入 `kill_dev_processes_for_workspace`
4. `.air.toml` 已加入仓库根目录，约束后端本地 watcher 的监听范围、延迟和错误行为
5. `README.md` 已补充本地开发说明，明确前端是 `vite`，后端是 `air` 自动编译重启

## 代码层结果

本次直接涉及的主路径文件如下：

- `gogogo.sh`
- `gogogo.1.sh`
- `gogogo.lib.sh`
- `.air.toml`
- `README.md`
- `tests/gogogo_air_config_test.sh`
- `tests/gogogo_dev_backend_command_test.sh`
- `tests/gogogo_dev_cleanup_wiring_test.sh`

其中值得单独记录的点有两个：

### 1. 保留了当前主工作区的 `9/platform` 接线

在整合 `local-dev-autoreload` 方案时，没有回退现有 `gogogo` 主菜单的 `9. AI 开发平台切换与 Symlink 管理` 能力，也保留了 `source "$SUB_SCRIPT" "$@"` 这类对子脚本参数透传的改动。

这意味着本次不是简单地把一个 feature worktree 强行 merge 回主工作区，而是按当前主工作区正在演进的 `gogogo` 体系做了语义融合。

### 2. 清理逻辑采用了 workspace 定向版本

主工作区里原先存在一版更粗暴的开发进程清理逻辑，特点是：

- 直接 `pkill -9`
- 直接按端口释放 `3000/5173`
- `air` 匹配方式较死

最终保留的不是这版实现，而是收敛后的版本：

- 不再依赖全局端口 kill
- 通过命令匹配 + 当前 workspace `cwd` 收敛清理范围
- 兼容 `air`、`bun run dev`、`vite` 和旧 `go run main.go`
- 先尝试正常结束，再对残留进程兜底 kill

这一步的原因很明确：当前仓库已经长期使用 worktree 并行开发，粗暴的全局端口释放容易误伤其他工作区或其他项目的本地进程。

## 验证事实

本次已完成以下验证：

### Shell 级验证

已运行：

```bash
bash -n gogogo.sh gogogo.1.sh gogogo.lib.sh tests/gogogo_air_config_test.sh tests/gogogo_dev_backend_command_test.sh tests/gogogo_dev_cleanup_wiring_test.sh tests/gogogo_session_name_test.sh
```

结果：通过，无语法错误。

已运行：

```bash
bash tests/gogogo_air_config_test.sh && \
bash tests/gogogo_dev_backend_command_test.sh && \
bash tests/gogogo_dev_cleanup_wiring_test.sh && \
bash tests/gogogo_session_name_test.sh
```

结果：

```text
PASS: .air.toml enforces backend-only autoreload rules
PASS: gogogo.1.sh uses air for backend and keeps vite for frontend
PASS: gogogo cleanup wiring is connected
PASS: session names differ
```

### 实机联调验证

除了静态 shell 检查外，本次还实际安装了 `air`，并做了真实 `tmux` 联调：

1. 安装 `air`
2. 启动 `./gogogo.sh dev`
3. 确认 tmux pane 状态为：

```text
0 air
1 bun
```

4. 通过临时 Go 文件两次触发 backend watcher
5. 两次都观察到 backend pane 出现新的启动日志与 `ready` 输出

这说明当前主工作区里的本地开发链路已经不是“理论上支持”，而是已经完成了真实自动编译重启验证。

## 取舍说明

在整合过程中，还专门比对了几个 worktree 对 `gogogo*` 的不同改动方向：

- `feature-gogogo-script` 的方向更偏向回退能力，包含固定 session、删除 `attach/stop`、弱化多 worktree 逻辑，这些都没有被采用
- `gogogo-deploy-github` worktree 因分叉点过早，并不代表独立的 `gogogo` 设计意图，因此没有作为合并依据
- 最终是以当前主工作区为基底，吸收 `local-dev-autoreload` 中与 `air` 和开发清理直接相关的部分

换句话说，本次变更不是“把一个 feature 分支原样抄回来”，而是围绕当前主工作区的真实演进方向做了一次定向融合。

## 结果

到本日志记录时，可以把当前事实概括为：

- 本地开发主入口仍然是 `./gogogo.sh dev`
- 前端开发态仍然是 `vite`
- 后端开发态已经切换为 `air`
- 后端语义不是运行时热替换，而是自动编译重启
- `gogogo stop/dev` 已经具备面向当前 workspace 的进程清理能力

这条日志只记录事实本身，不在这里上升为正式规范。若后续需要把“前端 `vite` + 后端 `air` + `gogogo` 主入口”收敛成长期规则，应在 `L3` 或 `L0` 层继续同步。
