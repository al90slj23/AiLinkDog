# gogogo 四屏本地开发工作台落地

## 背景

原先 `./gogogo.sh 1` 的本地开发模式只提供前后端双 pane，后续演进到三 pane 后，固定格式监控信息与持续事件流仍共用同一个下方 pane，导致两个问题：

- 固定格式信息屏容易被持续事件流挤压和污染；
- 事件流在 pane 高度不足时会出现可读性差、折行混乱和信息层级不清的问题。

同时，本地开发阶段还暴露出一系列运行时问题：

- 后端 `air` 会错误扫描前端目录，触发 `too many open files`；
- `stop` / `重启` 路径偶发残留后端孤儿进程，导致 `3000` 端口占用；
- 后端 ready 事件在“待确认地址 / 推断地址 / 明确地址”之间抖动，造成总结流重复刷屏；
- 单 monitor pane 难以同时承担“大屏”和“滚动流”两种职责。

## 本次事实变化

本次对 `gogogo` 本地开发工作流做了收敛，形成当前新的默认形态：

1. `./gogogo.sh 1` / `./gogogo.sh dev` 默认进入四 pane 本地开发工作台。
2. 上方左右两个 pane 保留原始日志：
   - 左侧后端 `air`
   - 右侧前端 `bun install && bun run dev`
3. 下方第一个长 pane 改为固定格式中文监控大屏。
4. 最下方新增一个持续总结流 pane，只负责滚动输出“人话事件”。

## 技术收敛

### 1. 后端 autoreload 规则固定为 `.air.toml`

引入并启用 `.air.toml`，明确 `air` 只监听后端 `.go` 代码，并排除：

- `web/`
- `node_modules/`
- `docs/`
- `.git/`
- `.ai/`

这样解决了早期 `air` 误扫前端依赖目录导致句柄爆炸的问题。

### 2. 运行时目录收敛到 `.tmp/gogogo/<session-suffix>/`

当前本地开发运行时文件统一写入：

- `backend.log`
- `frontend.log`
- `monitor.state`
- `monitor.events.log`

这让固定大屏和持续总结流可以共享同一份状态与事件来源，也让清理逻辑更明确。

### 3. 端口释放与孤儿进程处理增强

为了避免 `listen tcp :3000: bind: address already in use`，本次补强了两类处理：

- 清理 `air`、`vite`、`tmp/main` 以及 Go build cache 里的孤儿 `main` 进程；
- 在 `dev` 重启路径和统一启动前加入对 `3000` 端口释放的条件等待。

### 4. 后端地址与 ready 事件去抖

本次修正了 monitor 状态机：

- 后端 ready 事件允许先以“地址待日志确认”写入；
- 一旦从日志拿到明确 `http://localhost:3000/`，再刷新为确认地址；
- 一旦确认地址出现，不允许再被 `:3000` 这类推断信号降级回“推断地址”；
- 解决了持续总结流中 ready 事件反复刷屏的问题。

## 当前结果

当前 `gogogo` 本地开发入口已经不再只是“启动前后端”，而是一个明确分层的本地开发工作台：

- 原始日志：供调试和追根溯源
- 固定大屏：供一眼判断当前总体状态
- 持续总结流：供观察变化、告警和关键事件

这次变化已经属于长期执行工作流调整，后续相关规范与入口应以此形态为准。

## 后续注意项

- `gogogo.dev.monitor.sh` 当前已承担较多状态提取与事件处理职责，后续若继续扩展，应考虑拆分更细的状态机或解析模块；
- `gogogo.lib.sh` 也在持续积累运行时 helper，后续如果再增加本地开发相关控制逻辑，需要注意职责边界；
- 四 pane 布局的高度比例仍可继续微调，但“固定大屏”和“持续总结流”分离这一点已经确定，不应回退到单 pane 混合模式。
