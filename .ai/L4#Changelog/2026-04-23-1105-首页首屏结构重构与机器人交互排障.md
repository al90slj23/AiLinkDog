# 首页首屏结构重构与机器人交互排障

## 本次任务背景

本次工作围绕 `AiLink.Dog` 首页默认 landing page 持续推进，目标是把当前首页从旧版单屏 Banner 升级为更明确的品牌首页，并在过程中不断根据实际视觉效果与交互反馈做结构修正。

本轮任务重点不只是“把页面做出来”，而是反复校正以下几组关系：

1. 首屏主文案、指标、机器人主体之间的层级关系
2. 机器人背景、黄色光影背景、供应商块之间的分层关系
3. `Spline` 机器人与页面鼠标交互的实际命中链路
4. 首页 JSX 与样式文件的可维护性

在执行过程中，还穿插处理了一个与首页无关但会阻断前端构建的缺失页面组件问题，以确保首页改动可以被真实构建验证。

## 实际改动内容

### 1. 首页默认页从旧 Banner 切换为模块化 landing page

本次确认了 `HomePageContent` 覆盖逻辑继续保留，只有在未配置自定义首页内容时，才渲染新的默认 landing page。

相关事实：

- 保留了 iframe / markdown / html 自定义首页机制
- 默认首页改为由 `DefaultLandingPage` 负责拼装的模块化结构

### 2. 首屏模块职责进行了拆分

原先首屏逻辑集中在 `LandingHero.jsx` 一个文件中，后续持续调整极易互相干扰，因此本次将其拆成多个职责明确的组件：

- `LandingHeroCopy.jsx`
- `LandingHeroMetrics.jsx`
- `LandingHeroInfoCards.jsx`
- `LandingHeroCodeBlock.jsx`
- `LandingHeroAnimatedMetric.jsx`

拆分后，`LandingHero.jsx` 主要负责首屏布局编排、机器人懒加载和区块组合。

### 3. 首页样式从全局 `index.css` 抽离到 `home.css`

前期首页样式一直混在 `web/src/index.css` 中，随着首屏多轮试错，这部分样式定位和维护成本快速升高。本次把首页相关的 `ald-home-* / ald-ui-*` 样式抽离到：

- `web/src/pages/Home/home.css`

并在首页组件中显式引入，降低了后续局部样式误伤全局的风险。

### 4. 机器人资源从远端改为本地

原先 Spline 场景使用远端 `scene.splinecode`，在本地访问主页时出现：

- `ERR_CONNECTION_CLOSED`
- `Failed to fetch`

为降低这一链路的不稳定性，本次将场景下载到本地静态资源：

- `web/public/ald-robot.scene.splinecode`

此后首页不再依赖远端场景地址。

### 5. 为 `Spline` 区块补了失败兜底

考虑到本地资源、构建链或运行态仍有可能出错，本次为 `SplineScene` 增加了组件内错误边界与 fallback shell，避免机器人加载失败时把整个首页推入全局 `ErrorBoundary`。

### 6. 机器人后背供应商块经历了多轮位置与材质调整

在本次协作中，供应商块位置与材质反复校正，最终明确为：

- 位于机器人背景黄层和机器人主体之间
- 只在机器人区域的右半边出现
- 使用更偏磨砂、半透明、带轻高光边缘的玻璃片观感

此外，供应商数量也被扩充，形成更完整的兼容矩阵展示。

### 7. 首屏结构从三列误差态回收到更稳定的“左文案 / 右机器人”关系

本轮任务中，首屏结构曾多次在以下几种状态之间切换：

- 左中右三列，但视觉上出现“中间空带”
- 整个第一行背景铺满，但看起来像一个大框
- 左侧信息和机器人主体视觉上互相打架

最终判断是：

- 左边这组品牌与产品说明应该始终稳定落在机器人左侧
- 右边应保留机器人主体与供应商块的主视觉表达

虽然本轮仍在针对这块做持续调试，但当前结构已经明显从“大文件混乱试错态”收敛到更容易继续微调的状态。

### 8. 排查了“头部跟随鼠标转动”与“光晕只在左右边缘出现”的问题

本次通过 `Playwright` 实际检查了首页机器人区域的 DOM 命中情况，确认：

- `Spline` 实际渲染为 `canvas`
- 左侧和中间区域命中的最上层节点并不一致
- 供应商块层和背景层会影响鼠标事件真正落在哪一层

由此可以区分两类问题：

1. 页面上额外加的整机位移/旋转，不等于 Spline 模型自身的头部转动
2. 如果 `canvas` 没有拿到足够正确的鼠标事件，头部交互就不会恢复

本次已经删除了外层“整机跟手”的错误实现，并尝试把事件重新还给 `Spline` 本体，同时把光圈挂载点提升到更合理的首屏层级。

## 关键判断与边界

### 1. 这次的核心不是“调一个值”，而是先分清背景层与内容层

多轮来回的根因并不只是 `left/right/width` 数值不合适，而是：

- 背景层挂在哪个容器上
- 内容层是否压在背景之上
- 中间层是否会挡住鼠标事件

如果这些层级不先分清，单纯调数值只会反复出现“像一个框”“左右关系错位”“交互丢失”等问题。

### 2. Spline 交互问题不能用外层整机动画替代

用户真正关心的是：

- 机器人头部是否会随着鼠标转动

这件事不能用外层给整个机器人容器加 `translate3d / rotateX / rotateY` 来冒充。那种做法只会制造“整机在动”的假象，不能替代场景内部交互。

### 3. 供应商块的材质与尺寸必须服从“机器人背景的一部分”这个角色

这些块不是通用信息卡，不应完全套用其它模块的卡片语言；但它们也不能漂浮成轻飘飘的小标签。正确方向是：

- 保留矩阵秩序感
- 保留块尺寸感
- 材质升级为轻磨砂半透明玻璃片

### 4. 首页样式一旦成规模，就应该单独文件管理

本次从 `index.css` 中抽离首页样式，是后续稳定继续迭代的关键动作之一。否则后续每次调首页，都容易继续污染或误伤其他页面样式。

## 本次形成的开发经验

### 经验 1：视觉首页先分层，再排版

先定义：

- 背景层
- 中间层
- 内容层

再做左右列排版，会比边试边改更稳。

### 经验 2：交互异常优先做实际命中排查

当出现“左右两边能触发，中间不能触发”这种现象时，应优先检查：

- `elementsFromPoint`
- 实际命中的 DOM 树
- 事件最终落到哪一层

而不是继续猜 `z-index` 或继续微调 `left/right`。

### 经验 3：首页样式抽离后，修正效率明显提升

首页样式独立到 `home.css` 后，首屏相关问题的定位和修复比之前在 `index.css` 中查找快得多，也更不容易误伤全局样式。

### 经验 4：对视觉诉求的口头描述，必须先翻译成结构约束

例如：

- “在机器人左侧”
- “整行背景”
- “不是在一个框里”

这些话如果不先翻译成：

- 哪一层是背景层
- 哪一层是内容层
- 哪个容器负责宽度与定位

就会在实现中不断发生误解。

## 本次没有做的事情

1. 没有继续把首页样式再拆成更多子样式文件
2. 没有重新设计第二行的整体信息结构，只在首屏主结构上持续收敛
3. 没有替换 `Spline` 为纯本地 SVG/CSS 机器人方案，仍保留 Spline 版本
4. 没有继续处理构建体积偏大的问题，当前仍保留 `react-spline / physics` 大块产物
5. 没有把本次经验上升为正式 `L3` 规范，因为它们更像本次首页重构经验，而非当前项目必须遵守的长期硬规则

## 最终结果

本次工作结束后，首页默认 landing page 在工程组织和交互调试层面已经得到明显收敛：

1. 首页 JSX 已按职责拆分
2. 首页样式已从全局样式中抽离
3. 机器人场景改为本地资源
4. Spline 失败时有局部兜底
5. 供应商块位置、数量与材质进入可持续微调状态
6. 机器人交互问题已经从“猜测”推进到基于 Playwright 命中结果的实际排查

当前仍可继续优化的主要方向只剩：

- 首屏视觉关系的最终收口
- Spline 头部随鼠标转动的彻底恢复
- 首屏整体光影与层级的最终稳定
