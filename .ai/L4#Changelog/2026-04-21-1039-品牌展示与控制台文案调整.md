# 品牌展示与控制台文案调整

## 本次任务背景

本次工作起因于前端品牌展示不一致问题：

1. 浏览器强制刷新时，标签页会先短暂显示 `New API`
2. 用户已在系统设置中将站点名称设置为 `AiLink.Dog`
3. 浏览器控制台中的欢迎语与本地长期维护路线表达不一致

本次任务目标是：

- 排查强刷时标题闪回 `New API` 的根因
- 修复默认站点名与静态标题不一致问题
- 统一前端/后端默认品牌展示
- 在不破坏受保护上游标识的前提下，调整控制台与页脚中的品牌表达

## 根因排查结论

本次排查确认，强制刷新时标签页短暂显示 `New API` 的根因不是系统设置失效，而是“静态 HTML 标题”和“前端运行时标题”分属两套机制：

1. `web/index.html` 中存在静态 `<title>New API</title>`
2. React 启动后，前端才会通过 `getSystemName()` 和 `document.title` 在运行时覆盖标题
3. 强制刷新时，浏览器会先渲染静态 HTML，因此会先看到 `New API`
4. 当本地缓存为空时，`web/src/helpers/utils.jsx` 中的 `getSystemName()` 也默认回退到 `New API`

因此，这个问题的本质是默认值不一致，而不是系统设置读写异常。

## 实际代码改动

### 1. 统一静态标题与前端默认标题

修改以下文件：

- `web/index.html`
- `web/src/helpers/utils.jsx`

调整内容：

1. 将静态 HTML 标题从 `New API` 改为 `AiLink.Dog`
2. 将 `getSystemName()` 的默认回退值从 `New API` 改为 `AiLink.Dog`

效果：

- 强刷时不会再先闪 `New API`
- 本地缓存缺失时也会优先回退到 `AiLink.Dog`

### 2. 统一后端默认系统名

修改文件：

- `common/constants.go`

调整内容：

1. 将后端默认 `SystemName` 从 `New API` 改为 `AiLink.Dog`

效果：

- 前后端默认站点名进一步统一
- 在未显式配置站点名的默认场景下，也不会回退成 `New API`

### 3. 调整页脚品牌表达，但保留上游来源信息

修改文件：

- `web/src/components/layout/Footer.jsx`

调整内容：

1. 原页脚中的 `设计与开发由 New API` 改为 `来源项目 new-api / QuantumNous，AiLinkDog 本地维护`
2. 保留 `https://github.com/QuantumNous/new-api` 链接不变

效果：

- 不再把当前站点直接表现为 `New API` 品牌本体
- 同时保留上游来源与受保护标识

### 4. 调整管理页说明文案为中性表述

修改文件：

- `web/src/pages/Setting/Operation/SettingsChannelAffinity.jsx`

调整内容：

1. 将 `NewAPI 默认不会将入口请求的 User-Agent 透传到上游渠道...` 改为 `系统默认不会将入口请求的 User-Agent 透传到上游渠道...`

效果：

- 保留原说明语义
- 去掉不必要的产品名直写，降低管理页中的品牌干扰

### 5. 按专项 spec 校正浏览器控制台文案

涉及文件：

- `web/src/index.jsx`
- `docs/superpowers/specs/2026-04-21-console-branding-design.md`

本次过程中，曾短暂将原控制台欢迎语整体替换为 `AiLinkDog Local maintenance track | Upstream: ...`。

随后读取并比对已有专项 spec，确认该做法不符合既定边界。该 spec 明确要求：

1. 不删除、不替换现有 `WE ❤ NEWAPI` 欢迎语
2. 不弱化 `new-api / QuantumNous` 受保护标识
3. 仅允许在原欢迎语之后追加一条 `AiLinkDog` 说明

因此最终将控制台输出调整为两段：

1. 保留原始欢迎语：

```text
WE ❤ NEWAPI Github: https://github.com/QuantumNous/new-api
```

2. 追加本地维护路线说明：

```text
ALD AiLinkDog
Originated from NEWAPI, AiLinkDog is our long-term maintenance and evolution track, continuously iterated and refined around real-world operational needs.
```

最终结果是：

- 上游受保护欢迎语继续保留
- 本地长期维护路线身份得到补充表达
- 控制台文案与专项 spec 保持一致

## 关键判断与边界

本次工作中明确了以下边界：

1. `new-api` 与 `QuantumNous` 属于受保护来源标识，不能为了本地品牌统一而整体替换
2. 用户可见品牌默认值、静态标题、管理说明文案可以按本地站点需要做最小调整
3. 控制台欢迎语属于边界敏感区，必须遵循已有专项 spec，而不是临场重写
4. 页脚这类展示位更适合采用“双表达”：既保留来源项目，又说明 `AiLinkDog` 是本地维护路线

## 本次没有做的事情

本次明确没有继续处理以下内容：

1. 没有修改 `About` 页面中关于上游仓库、版权和来源关系的说明
2. 没有修改 `web/index.html` 中的 `meta name="generator" content="new-api"`
3. 没有改动任何 `github.com/QuantumNous/new-api/...` 的 Go import path
4. 没有补动 `common/init.go` 中启动输出与 CLI 帮助文案
5. 没有同步 `L1-L3`、`L5` 记忆层，因为本次未形成新的长期规则，只是按现有规则落地

## 与 skill 规则相关的额外收尾

在本次收尾过程中，还识别出项目 skill `standards-sync-on-complete` 的口径与当前协作要求不一致。

原 skill 将 `L4` 工作日志写成“按条件判断是否补写”；但当前项目实际要求是：

1. 只要进入完成态总结或收尾
2. `L4#Changelog` 必须补写一条工作日志
3. `L0-L3` 与 `L5` 才是按需判断是否同步

因此本次同步调整了该 skill，明确：

- `L4` 为完成态必写
- `L5` 仅在确有长期知识需要沉淀时才更新
- `L0-L3` 继续按长期规则变化决定是否同步

## 最终结果

本次任务完成后，当前仓库在品牌展示层面的状态如下：

1. 强制刷新时标签页不再先闪 `New API`
2. 前后端默认站点名统一为 `AiLink.Dog`
3. 页脚保留上游来源信息，同时明确 `AiLinkDog` 本地维护身份
4. 管理侧说明文案减少了不必要的品牌名暴露
5. 浏览器控制台同时保留上游欢迎语与本地长期维护说明
6. 完成态 skill 已同步为“每次总结必须生成 `L4` 工作日志”的项目规则口径
