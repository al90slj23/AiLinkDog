# 2026-04-22 zm_api 导入与邀请返利增强版进入上线前收尾

## 背景

在前一阶段已经完成邀请返利增强版双页面、数据库扩展域、自动分佣主链和高强度验证计划之后，本轮工作继续推进两条主线：

1. 将 `zm_api.sql` 中的旧 MySQL 业务数据正式导入当前项目使用的 PostgreSQL `ailindog`。
2. 在导入后，让当前项目的新邀请返利功能与旧数据、旧配置和新扩展域进入可验收状态。

本次工作同时涉及：

1. legacy 数据导入。
2. referral 新域的保守兼容补建。
3. 旧运行配置与当前前端菜单结构的兼容。
4. 邀请返利核心链路与数据库真实验证。
5. 前端 React 18 StrictMode + Semi UI warning 的持续清理。

## 本次完成的关键事实

### 一、MySQL 旧库已导入 PostgreSQL

本次已经正式执行：

1. `zm_api.sql` 的 legacy 25 张业务表导入当前 PostgreSQL。
2. 导入前已对 `ailindog` 做完整备份。
3. 导入后已执行：
   - schema check
   - sequence reset
   - postcheck

当前判断：

1. legacy 数据已成功承接进 PostgreSQL。
2. 旧业务表主键和唯一约束未发现新的导入冲突。
3. 数据库已经从“当前项目自有样本数据”切换为“旧库主数据 + 当前扩展域”的状态。

### 二、referral 新域采用保守回填策略

由于旧 `zm_api.sql` 本身不包含：

1. `referral_plans`
2. `referral_accounts`
3. `referral_links`
4. `referral_commissions`
5. `referral_withdrawals`

因此本次没有尝试伪造历史 referral 事件流水，而是采用保守策略：

1. 保留当前 PostgreSQL 中的默认 referral 方案结构。
2. 清掉之前为了功能验证写入的 referral 样本脏数据。
3. 仅按旧 `users.inviter_id` 等真实邀请关系痕迹，补建 `referral_accounts`。
4. 不伪造历史：
   - 邀请链接
   - 佣金流水
   - 提现记录

当前结果：

1. `referral_accounts` 已保守回填出 6 条记录。
2. `referral_links` / `referral_commissions` / `referral_withdrawals` 在导入后干净保留为空或当前阶段的真实新数据状态。

### 三、旧全局配置与当前侧边栏结构的兼容问题已修

导入 `options` 表后，旧库中的：

1. `SidebarModulesAdmin`
2. `DrawingEnabled`

等配置会直接影响当前项目运行态。

本次已明确并处理：

1. `SidebarModulesAdmin` 导入后会把当前项目新增的菜单项隐藏。
2. 通过兼容逻辑和手动重置，已恢复当前项目完整默认侧栏模块配置。
3. `DrawingEnabled` 已恢复为当前项目默认开启状态。

这使得：

1. `邀请返利中心`
2. `邀请返利管理`
3. 聊天区、任务、绘图等菜单

在当前项目里重新进入可见和可验收状态。

### 四、邀请返利核心链路已真实打通并验证

本次继续确认并验证了邀请返利核心业务链：

1. 管理员新增 / 编辑 / 启停返利方案。
2. 用户首次锁定返利方案。
3. 用户创建邀请链接。
4. 用户充值后自动生成一级 / 二级返利。
5. 返利真实写入：
   - `referral_commissions`
6. 返利真实同步更新：
   - `users.aff_quota`
   - `users.aff_history`
7. 用户发起提现申请。
8. 管理员查看并审核提现。

已通过数据库样本验证：

1. A -> B -> C 两级邀请关系成立。
2. C 充值 100 后：
   - B 生成一级返利
   - A 生成二级返利
3. 佣金记录和旧用户返利字段同步更新。

### 五、管理页统计口径与默认方案初始化已收平

本轮已修复：

1. 默认方案重复初始化。
2. 默认方案初始化误删管理员自定义方案。
3. 管理页顶部统计卡口径错误。
4. 管理页概览与返利方案职责重复。

这使得管理页当前更接近：

1. 概览只展示统计和方案运营摘要。
2. 返利方案负责新增 / 编辑 / 启停。

### 六、前端 warning 清理已进入逐页收尾阶段

本轮还并行处理了多个页面在 React 18 StrictMode 下的 warning：

1. `/console/token`
2. `/console/channel`
3. `/console/models`
4. `/console/user`
5. `/console/setting?tab=model-deployment`
6. `/console/setting?tab=operation`
7. `/console/subscription`

当前采用的统一策略是：

1. 保留 `StrictMode`。
2. 不通过关闭检查来掩盖问题。
3. 用项目级 `AppDropdownMenu` 替代 Semi `Dropdown`。
4. 将 `Tooltip/Popover` 的触发器从 `Tag/Button` 改成更安全的 `span/div` 包裹。
5. 修正不合法 props，例如：
   - `Tabs size='default'`
   - `Button type='outline'`
   - `Card hoverable`

当前判断：

1. 这部分已经不是完全没动。
2. 已进入“根据最新 warning 栈逐页清理”的收尾阶段。
