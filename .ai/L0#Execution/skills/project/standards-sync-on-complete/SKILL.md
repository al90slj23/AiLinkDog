# Standards Sync On Complete

## 概述

这个 skill 用于在任务完成前，先补写一条 `L4` 工作日志记录本次任务事实。随后再判断本次改动是否引入了长期规则变化，并据此决定 `L3`、`L2`、`L1`、`L0` 以及 `L5` 是否需要同步。

核心原则：`L4` 是事实日志层，不是纯化目标层。只要进入完成态总结或收尾，就应新增一条对应的工作日志；即使和前一天任务存在事实重叠，也照常按任务完成态记录。规则层与执行层才需要按需持续收敛。

## 何时使用

在你准备宣称“任务完成”之前，如果出现以下任一情况，就必须调用本 skill：

- 用户要求“总结”“收尾”“归档”“补工作日志”“同步规范”
- 你准备给出完成态总结
- 本次改动可能涉及长期规则变化、执行方式变化或记忆层同步

调用本 skill 后，无论本次任务大小、是否只是实现层修改、是否涉及长期规则变化，都必须补写一条新的 `L4#Changelog` 工作日志。

另外，如果本次改动涉及以下任一情况，还必须继续检查 `L0-L3` 与 `L5` 是否需要同步：

- 正式开发规范变化
- 架构边界或分层约束变化
- 数据库方向、数据库结构约束或持久化规则变化
- API 设计约束变化
- 命名规则变化
- AI 记忆体系分层规则变化
- 平台接入方式变化
- `L0` 执行工件使用规则变化

以下情况通常不需要额外同步 `L0-L3` 与 `L5`，但仍需要补写 `L4` 工作日志：

- 纯实现代码修改，且不改变长期规则
- 纯 bug 修复，且不改变现行规范
- 纯测试补充，且不改变规则口径
- 纯 UI 或文案调整，且不改变协作流程或项目规则

### 自然语言触发方式

除了直接点名 `standards-sync-on-complete`，以下自然说法也应触发本 skill：

强触发：

- “收尾一下”
- “整理一下这次改动”
- “补一下工作日志”
- “同步一下规范”
- “看看这次要更新哪些记忆层”
- “归档一下”

弱触发：

- “总结一下”
- “归纳一下”
- “更新下”

其中弱触发的含义可能只是口头总结；如果上下文同时出现：

- 工作日志
- 规范同步
- 记忆层更新
- 收尾

则默认进入本 skill，而不是只做口头概括。

## 流程

### 1. 先确认 `L4` 工作日志必须补写

先明确执行以下硬规则：

- 只要本 skill 被用于任务完成态收尾，`L4#Changelog` 必须补写一条本次工作的事实日志
- 这里不存在“要不要写 L4”的判断，也不需要做去重判断；同主题或相邻任务存在事实重叠是正常现象

日志内容至少应覆盖：

- 本次任务背景
- 实际改动内容
- 关键判断与边界
- 最终结果与未做事项

### 2. 再判断是否存在长期规则变化

先明确回答：

- 本次改动是否只是实现层变化？
- 还是已经改变了“团队以后应该怎么做”的规则？

如果答案是否定的，直接输出：

`本次改动未识别出长期规则变化；除 L4 工作日志外，不触发其他记忆层纯化。`

不要为了“显得完整”强行升级文档。

### 3. 读取 `L4` 作为事实依据，并形成工作日志

无论是否识别出长期规则变化，都要读取相关 `L4#Changelog`、当前代码改动和已有规范，形成一条新的 `L4` 工作日志。

这里的目标不是纯化 `L4`，而是借助 `L4` 回答：

- 这次变化的事实是什么
- 以前为什么这样规定
- 当前是否已经出现了新旧规则切换

这里的“近期工作总结”统一按“工作日志”理解，默认归入 `L4#Changelog`。

补写 `L4` 工作日志时，应遵循 `L4` 目录现有命名规范：

- 使用“时间戳 + 主题”命名
- 推荐格式：`YYYY-MM-DD-HHMM-主题.md`
- 不使用 `recent-*`、`latest-*` 这类滚动语义文件名

### 4. 检查本次改动相关代码文件的行数

对本次改动直接触达的代码文件做行数检查。

目标不是机械卡阈值，而是辅助判断：

- 是否已经出现明显过大的代码文件
- 文件规模是否已经影响可维护性、边界清晰度或后续协作
- 是否需要把这类问题上升到 `L3` 规则或 `L0` 执行工件中

至少输出：

- 文件路径
- 当前总行数
- 是否需要关注
- 可能影响的记忆层

不要做全仓库无差别扫描，只看本次改动直接涉及的代码文件。

## 5. 检查真正需要升级的层

除必须补写的 `L4` 外，只检查以下层：

- `L3#Standards`
- `L2#Index`
- `L1#Overview`
- `L0#Execution`

### L3

检查：

- 当前正式规则是否已经过时
- 是否已有同主题规范文件需要改写
- 是否需要新增规则文件
- 多个 `L3` 文件之间是否出现口径矛盾

### L2

检查：

- `toc.md` 或相关索引是否能把读者导向正确规范
- 新规则是否缺少入口
- 旧入口是否仍指向过时规则

### L1

检查：

- 本次变化是否提升为项目级总规则
- 是否影响项目总览、主路线、核心边界
- 是否需要更新 `guide.md` 或根级协作入口文件

### L0

检查以下执行工件是否需要同步：

- `skills/`
- `workflows/`
- `hooks/`
- `templates/`

重点判断：执行入口、操作说明、自动化载体是否还与当前规则一致。

### L5

检查：

- 本次任务是否形成了值得复用的项目知识、经验模式或排障结论
- 这些内容是否已经不适合放在 `L4` 的事件型日志里，而需要沉淀为知识层
- 是否真的需要新增/更新 `L5`，而不是把一次性事实误写成长期知识

## 6. 输出待确认项

在真正改文档之前，必须先给出结构化结论。

输出至少包含：

- `L4` 工作日志补写结论（必须补写）
- 是否识别出长期规则变化
- 代码文件行数检查结果
- 需要纯化或升级的层
- 每层触发原因
- 建议修改动作
- 需要用户确认的事项

推荐格式：

```md
## 规范同步判断

- L4 工作日志：必须补写
- 是否涉及长期规则变化：是 / 否

## 代码文件行数检查

- <文件路径>：<行数> 行；<是否需要关注>；<影响层>

## 建议更新层

- L5：<原因>
- L3：<原因>
- L2：<原因>
- L1：<原因>
- L0：<原因>

## 需要用户确认的事项

- <确认项 1>
- <确认项 2>
```

## 7. 未确认前不要自动同步

这是强约束：

- 不要自动修改存在取舍空间的规则文档
- 不要假设用户已经同意升级到 `L1`
- 不要把 `L4` 当成需要“消冲突”的层
- 不要把“近期工作总结”写去 `L5`

如果已经能明确判断只是缺一个索引入口、没有任何歧义，也应先在输出中说明，再等待确认。

唯一例外：如果用户已经明确要求“每次总结必生成一个工作日志”，则 `L4` 工作日志不需要再二次确认是否要写，而是直接纳入完成动作；需要确认的只剩其他层是否同步。

## 8. 确认后再按层同步

得到用户确认后，再同步对应层。

推荐顺序：

1. 先补 `L4` 工作日志（必做）
2. 判断并更新 `L5`（仅在确有长期知识需要沉淀时）
3. `L3` 再定正式规则
4. `L2` 再补索引与入口
5. `L1` 再决定是否升级为项目级总规则
6. `L0` 最后同步执行工件

## 常见错误

| 错误想法 | 正确做法 |
|---------|---------|
| 这次只是小改动，不用写 `L4` | 错。只要进入完成态总结，`L4` 工作日志必须补写 |
| 没有规则变化，所以不用写工作日志 | 错。没有规则变化只意味着通常不动 `L0-L3` / `L5`，不意味着可以跳过 `L4` |
| 和昨天内容有重复，所以这次 `L4` 可以省略 | 错。`L4` 记录的是本次任务完成态事实，允许和其他任务日志有重叠 |
| `L4` 里前后说法不同，需要纯化 | 不需要。`L4` 记录的是阶段性事实，只要真实即可 |
| 近期工作总结应该继续写到 `L5` | 在当前体系下，这类持续追加型记录应归入 `L4` 工作日志 |
| `L4` 工作日志可以叫 `recent-*` | 不应这样做。`L4` 应使用时间戳 + 主题命名，保持事件型日志结构 |
| 文件行数只是实现细节，不必纳入完成态判断 | 如果文件规模已影响维护边界，就应该进入完成态检查输出 |
| 这次规则变化很明显，直接改完再说 | 先列出影响层和待确认项，再等确认 |
| 只改 `L3` 就够了 | 同时检查 `L2`、`L1`、`L0` 是否脱节 |
| `L0` 只看 skill | 还要检查 `workflows/`、`hooks/`、`templates/` |

## 红旗

出现以下念头时，说明你在跳过本 skill 应有的判断：

- “这次只是个小任务，不用写 `L4`”
- “没有长期规则变化，所以 `L4` 也可以不写”
- “和昨天有重复，所以今天这条 `L4` 可以省略”
- “这只是顺手改个规范，不用同步其他层”
- “L4 里有旧说法，顺手清掉更干净”
- “近期工作总结先继续丢到 L5 再说”
- “文件行数只是代码细节，不用在完成时判断”
- “先把文档都改了，再告诉用户”
- “只改执行入口，不用改正式规范”

这些都意味着你应该停下来，重新执行本 skill 的分层检查流程。
