# 质量规范：测试与验证

## 原则

在声称“已完成”“已修复”“已通过”之前，必须先运行对应验证命令。

验证结论必须以 fresh 输出为依据，而不是凭印象或上一次运行结果。

出现报错或 warning 时，必须先看当前仓库中的对应关联文件，再去对照上游 `new-api` 官方仓库中的同名文件，判断：

- 上游是否已经修复或调整；
- 当前仓库是否落后于上游；
- 问题是否由本地改动引入；
- 还是上游本身也存在同类问题。

不得在没有做当前仓库 / 上游文件对照的情况下，先入为主地假设问题一定来自本地修改或一定来自上游。

## 当前仓库常用验证

- `go test ./...`
- `go test ./relay/helper ./relay/channel/claude`
- `cd web && bun run build`
- `./gogogo.sh test`
- `./gogogo.sh build`
- `./gogogo.sh lint`

## 注意

`lint` 失败不一定代表功能错误，也可能只是格式或规范问题；但必须如实报告。

`build` 失败也不一定意味着脚本逻辑错误，可能是依赖安装或构建环境问题；同样必须基于输出如实报告。
