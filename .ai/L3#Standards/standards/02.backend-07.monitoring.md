# 02.backend-07.monitoring

## 适用范围

本规则适用于 AiLinkDog 内部的监控与状态中心能力，包括：

- 平台内置监控
- 用户自定义监控
- 监控探测执行
- 状态聚合
- 事件记录
- 监控扣费记录

## 规则

### 1. 监控目标不得直接落入 `channels` 表

监控目标必须使用独立监控域模型存储，例如：

- `monitor_targets`
- `monitor_runs`
- `monitor_status_snapshots`
- `monitor_events`
- `monitor_billing_records`

原因：

- `channels` 表属于业务转发实体
- 监控目标需要额外承担归属、可见性、频率、私有 key、扣费等职责
- 直接复用 `channels` 表会污染业务渠道语义

### 2. 监控目标类型复用现有 `channel type` 协议体系

虽然监控目标不能直接复用 `channels` 表，但监控执行时应尽量复用：

- 现有 `channel type` 枚举
- 现有协议兼容能力
- 现有 relay/channel 请求适配思路

不要为监控系统重新发明第二套 provider/type 体系，除非现有协议栈无法复用。

### 3. 用户自定义监控默认必须私有

对于 `source_type = custom` 的监控目标：

- `owner_user_id` 必填
- `visibility` 必须为 `private`
- 公开摘要与公开事件查询必须额外限制 `source_type = platform`

不允许通过脏数据、手工写库或后台扩展使用户自定义监控进入公开状态页。

### 4. 用户监控密钥必须基于 `CryptoSecret` 加密存储

用户提交的监控密钥必须遵守：

- 存库前加密
- 列表与详情不回显明文
- 编辑页不回显旧 key
- `CryptoSecret` 为空时拒绝加密和解密

不允许以空串或默认值派生固定密钥继续写库。

### 5. 用户监控扣费必须与记账保持一致性

用户自定义监控执行时：

- 扣费前应校验余额
- quota 扣减应使用条件更新，避免并发下超扣
- `run / snapshot / event / billing` 的落库与记账应保持在同一事务边界内
- 余额不足时不应继续正常执行探测

### 6. 状态中心必须优先提供最小真实动态链路，而不是静态假页面

只要进入“状态中心已实现”阶段，就必须优先打通：

- 后端 monitor API
- 前端状态页动态数据源

不应长期停留在只有静态文案和占位骨架的状态页。
