# 工具规范：gogogo

## 定位

`gogogo.sh` 是当前仓库的统一脚本入口，用于本地开发、构建、测试、检查、清理和部署。

## 当前支持命令

- `dev`
- `build`
- `test`
- `lint`
- `clean`
- `deploy`
- `platform`
- `attach`
- `stop`

## 使用原则

- `dev` 是当前仓库的本地开发主入口，默认通过 `tmux` 同时拉起前后端开发环境
- `dev` 的前端 pane 使用 `bun install && bun run dev`，由 `vite` 提供 HMR 或自动刷新
- `dev` 的后端 pane 使用 `air`，语义是“检测 `.go` 文件变化后自动编译重启”，不是真正的运行时热替换
- `stop` 与 `dev` 内部重启路径都需要按当前 workspace 清理相关本地开发进程，避免残留 `air`、`vite`、旧后端进程或旧 tmux 会话干扰新会话
- `platform` 用于统一进入 AI 开发平台切换与 symlink 管理能力
- `test` 用于统一运行后端测试
- `build` 用于统一触发前端构建与后端编译
- `lint` 用于反映前端当前格式/规范状态
- `clean` 需要保留 `web/dist/index.html` 占位文件，避免影响后端 embed
- `deploy` 用于统一进入部署相关脚本能力，但不改变当前仓库以本地开发和常规构建验证为主的工作流
- `gogogo.sh 7` 的工作日志输出目标当前统一收敛到 `L4#Changelog/`
- `gogogo.sh 7` 生成并保存总结后，当前主流程应直接结束，不再默认进入“打开总结报告和纯化目标文件”的 follow-up 菜单
- `gogogo` 相关 AI provider 在大 diff / 大 prompt 场景下必须先做上下文收敛，不能无上限把完整 diff 直接拼进 provider 输入
- 即使脚本仍可能涉及 Docker，Docker 也不是当前项目的正式主工作流或推荐主入口
