# 质量规范：协作与 PR 流程

## 来源

当前规则主要来自以下协作文档：

- `.github/PULL_REQUEST_TEMPLATE.md`
- `.github/ISSUE_TEMPLATE/bug_report.md`
- `.github/ISSUE_TEMPLATE/feature_request.md`

这些文件虽然面向 GitHub 页面填写，但其中表达的是当前仓库真实的协作纪律，应进入长期主记忆。

## PR 描述规则

- PR 摘要应由提交者人工整理，不应直接粘贴未经处理的 AI 输出
- 描述应解释做了什么，以及为什么这样修改有效
- 变更说明应基于对代码逻辑的理解，而不是表面复述

## 提交前要求

- 提交前应检查现有 Issues 与 PR，避免重复提交
- 变更范围应聚焦当前任务，不应混入无关改动
- 提交前应完成本地验证，并能提供复核证据
- 提交内容不得包含敏感凭据

## Bug 与 Feature 的区分

- `Bug fix` 不应被滥用来包装设计取舍、理解偏差或预期不一致
- 重大新功能更适合先通过 Issue 沟通，再进入实现
- 如果是 bug fix，当前模板倾向于要求存在对应 Issue 或关联记录

## Issue 填写规则

- 问题反馈与功能请求都要求遵循模板，不应随意删空模板结构
- 提交前应先看文档、README 和现有问题列表
- 不符合模板要求、明显重复或低质量的问题描述，存在被直接关闭的风险

## 当前仓库协作风格

- 倾向于要求提交者先自行做功课，再提出问题或提交变更
- 倾向于要求变更说明简洁、真实、可复核
- 倾向于把“是否真的理解修改内容”作为协作质量的一部分

## 与 .ai 的关系

- 这些规则应作为 `.ai` 中质量规范的一部分长期保留
- 当 GitHub 模板发生明显变化时，应同步检查本文件是否需要更新
