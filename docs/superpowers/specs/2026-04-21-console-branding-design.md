# Console Branding Design

## 背景

当前前端入口 `web/src/index.jsx` 已在浏览器控制台输出受保护的 `new-api` 欢迎语：

```text
WE ❤ NEWAPI Github: https://github.com/QuantumNous/new-api
```

根据当前仓库知识与规范，`AiLinkDog` 是一个基于 `new-api` 的本地长期维护与演进路线。它强调的是围绕真实运营需求持续迭代、优化与调整，但不能替代或覆盖上游 `new-api` / `QuantumNous` 的来源身份与受保护标识。

本次需要在不触碰受保护欢迎语和不扩大改动范围的前提下，为控制台补充一条 `AiLinkDog` 的身份说明。

## 目标

本次设计的目标只有一个：在浏览器控制台中新增一条 `AiLinkDog` 说明，准确表达以下语义：

- `AiLinkDog` 起源于 `NEWAPI`
- `AiLinkDog` 是当前仓库的长期维护与演进路线
- 该路线会围绕真实运营场景持续迭代和调整

## 非目标

本次明确不做以下事情：

- 不删除、不替换现有 `WE ❤ NEWAPI` 欢迎语
- 不修改与弱化任何 `new-api` / `QuantumNous` 受保护标识
- 不处理 React DevTools 控制台提示
- 不处理 `code-inspector-plugin` 的开发期提示
- 不把新增说明扩展到页面 UI、About 页面或首页
- 不增加环境变量、开关项或新的配置机制

## 方案选项

### 方案 A：追加一条纯英文说明

在现有 `NEWAPI` 欢迎语之后追加一条英文控制台说明。

优点：

- 风格与现有控制台信息一致
- 对外部协作者更直接

缺点：

- 对本地维护语义的表达略显生硬

### 方案 B：追加一条中英结合说明

在现有 `NEWAPI` 欢迎语之后追加一条以 `ALD AiLinkDog` 为标题、正文为英文说明的补充信息。

优点：

- 同时兼顾本地维护语境与对外可读性
- 能完整表达“源于 `NEWAPI`、长期维护、按真实运营持续演进”
- 与现有欢迎语形成“上游来源 + 本地长期路线”的并列关系

缺点：

- 控制台信息会多一行，信息量略增

### 方案 C：把现有欢迎语改写成一段完整介绍

直接把当前控制台欢迎语扩展成包含 `new-api` 与 `AiLinkDog` 的复合文案。

优点：

- 信息集中，输出位置单一

缺点：

- 会直接触碰受保护欢迎语附近的边界
- 后续维护上游与本地身份关系时更容易混淆
- 改动侵入性高于“追加一条说明”

## 方案选择

选择方案 B。

理由如下：

- 能在不改动原欢迎语的前提下补充 `AiLinkDog` 定位
- 能清楚区分“上游来源身份”和“本地长期维护路线”
- 改动最小，只需要在现有入口文件中追加一条控制台输出

## 设计细节

### 输出位置

沿用现有控制台欢迎语的落点，直接在 `web/src/index.jsx` 中追加新的 `console.log(...)`。

不新增 helper、不新增配置、不新增组件。

### 输出结构

控制台最终保留两段语义：

1. 现有上游来源欢迎语

```text
WE ❤ NEWAPI Github: https://github.com/QuantumNous/new-api
```

2. 新增本地长期维护路线说明

```text
ALD AiLinkDog
Originated from NEWAPI, AiLinkDog is our long-term maintenance and evolution track, continuously iterated and refined around real-world operational needs.
```

### 样式原则

新增说明需要遵循以下样式原则：

- 视觉权重低于现有 `NEWAPI` 主欢迎语
- 与现有绿色系风格协调，但不抢主标题
- 更偏说明性文字，而不是品牌主标语

建议的视觉关系是：

- 第一条强调上游来源与识别度
- 第二条强调本地维护定位与补充说明

### 文案边界

新增文案必须满足以下边界：

- 明确 `AiLinkDog` 源于 `NEWAPI`
- 明确 `AiLinkDog` 是长期维护与演进路线
- 明确其迭代依据是现实运营需求
- 不得把 `AiLinkDog` 表述成替代 `new-api` 身份的新项目

## 影响范围

本次变更影响范围极小：

- 仅修改 `web/src/index.jsx`
- 仅影响浏览器控制台输出
- 不影响页面渲染、路由、状态管理、国际化和业务逻辑
- 不影响前端构建产物的业务行为

## 风险与处理

### 风险 1：文案越界，弱化上游来源关系

处理方式：

- 保留原始 `NEWAPI` 欢迎语不变
- 新增说明第一句明确写出 `Originated from NEWAPI`

### 风险 2：控制台信息变多，显得冗余

处理方式：

- 只追加一条说明
- 不在本次顺带处理其他控制台提示

## 验证方式

实现后需要完成以下验证：

1. 启动前端开发环境，确认控制台仍先显示现有 `WE ❤ NEWAPI ...` 欢迎语
2. 确认控制台出现新增的 `ALD AiLinkDog` 说明
3. 确认页面功能、路由与初始化行为未受到影响
4. 在 `web/` 目录执行 `bun run build`，确认前端构建通过

## 结论

本次设计采用“保留上游欢迎语、追加本地路线说明”的最小改动方案，在不触碰受保护标识的前提下，为控制台补充 `AiLinkDog` 的长期维护与演进定位。
