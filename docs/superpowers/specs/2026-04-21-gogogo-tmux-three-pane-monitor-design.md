# gogogo.sh 本地开发三屏监控设计

## 背景

当前仓库的本地开发主入口是仓库根目录下的 `./gogogo.sh 1` 或 `./gogogo.sh dev`。该入口会在 `gogogo.1.sh` 中通过 `tmux` 创建一个开发会话，并将后端与前端分别放在两个 pane 中启动。

当前行为已经满足“前后端一起启动并能直接看日志”的基本诉求，但仍有几个明显缺口：

- 只看到原始日志，缺少一眼能识别的摘要状态
- 启动完成后，需要开发者自己从滚动输出里判断访问地址、是否就绪、启动花了多久
- 当前 pane 里出现 warning、error、端口变化或重启等事件时，没有统一的人话提示
- 前后端都在持续输出，重要变化容易被普通编译日志或依赖日志淹没

用户希望保留 `tmux` 作为本地开发入口形态，但把现有双屏升级为三屏：

- 一个 pane 继续显示后端原始输出
- 一个 pane 继续显示前端原始输出
- 新增一个 pane 专门以“人话摘要”的方式持续监控前两个 pane 的输出，并显示关键状态

## 目标

本次设计聚焦本地开发体验优化，不改变应用业务逻辑与生产运行方式。

具体目标如下：

- 保持 `./gogogo.sh 1` 作为本地开发主入口
- 保持 `tmux` 会话作为开发态主承载方式
- 将原有双 pane 升级为三 pane 布局
- 第三个 pane 持续展示“固定看板 + 事件流”的监控摘要
- 监控摘要能够从前后端输出中提取前端地址、后端地址、启动耗时、状态变化、警告与错误
- 监控逻辑与前后端启动逻辑解耦，避免把摘要解析塞进业务启动命令本身
- 日志、状态文件和 `tmux` 会话均按工作区隔离，避免不同 worktree 互相污染

## 非目标

本次设计明确不做以下事情：

- 不改动 Go 或 React 业务代码来专门输出监控摘要
- 不引入额外 GUI 控制台、Web Dashboard 或浏览器页面
- 不把监控 pane 做成完整日志分析平台
- 不在第一阶段支持复杂规则引擎、通知推送或历史持久化报表
- 不改变 `gogogo.sh` 的其他菜单项行为
- 不依赖 `tmux capture-pane` 作为主数据源

## 现状判断

### 当前脚本结构

从现有脚本可以确认：

- `gogogo.sh` 是主入口，负责菜单、参数解析与分发
- `gogogo.1.sh` 负责本地开发模式
- `gogogo.lib.sh` 负责 session 名、命令校验、进程清理等共享逻辑
- 当前 `gogogo.1.sh` 会创建名为工作区专属的 `tmux` session，并启动两个 pane：后端 `air`、前端 `bun install && bun run dev`

### 当前测试约束

现有 shell 测试已经验证以下事实：

- 后端 pane 必须启动 `air`
- 前端 pane 必须继续执行 `bun install && bun run dev`
- 本地开发启动前与停止时都要清理当前工作区相关进程

这意味着三屏改造不能破坏既有主命令语义，但需要同步更新测试，使测试不再默认假设只有两个 pane。

## 方案选项

### 方案 A：直接读取 `tmux` pane 输出

第三个 pane 自己周期性调用 `tmux capture-pane`，读取前后端 pane 缓冲区内容，再在内存中分析状态。

优点：

- 表面上不需要额外日志文件
- 对前后端启动命令改动较少

缺点：

- 依赖 `tmux` pane 缓冲区，长输出容易截断
- 对 pane 索引、布局、滚屏长度比较敏感
- 监控逻辑与 `tmux` 细节耦合较深，稳定性不足
- 不利于测试与后续扩展

### 方案 B：前后端输出同步写入日志，第三个 pane 运行独立监控脚本

前后端 pane 仍然直接显示原始输出，但同时将输出落到工作区专属日志文件。第三个 pane 运行独立 shell 监控脚本，解析这些日志并渲染看板与事件流。

优点：

- 数据源稳定，不依赖 `tmux` 缓冲区是否还可见
- 前后端 pane 仍然保留熟悉的原始日志体验
- 监控逻辑与布局逻辑解耦，便于测试和迭代
- 后续新增匹配规则、状态文件或更多摘要项都更容易

缺点：

- 需要新增日志目录与监控脚本
- 启动命令需要包装成“显示输出并同时落日志”的形式

### 方案 C：第三屏只显示静态信息和进程存活状态

第三个 pane 不持续分析日志，只显示预设地址、进程是否存活、启动命令是否发出。

优点：

- 实现最快
- 对日志解析要求最低

缺点：

- 不能回答“发生了什么变化”
- 无法提炼 warning、error 或端口变化
- 不符合用户希望“持续监控左边两个屏输出”的核心诉求

## 方案选择

选择方案 B：前后端输出同步写入日志，第三个 pane 运行独立监控脚本。

选择原因如下：

- 这是最稳妥的持续监控方案，能真实长期跟踪前后端输出
- 能最大程度保留现有前后端 pane 的直接观察体验
- 监控 pane 的数据来源稳定，避免 `tmux capture-pane` 的截断和依赖问题
- 可以在不改业务代码的前提下做出“人话摘要”
- 便于通过 shell 测试验证脚本连线关系

## 设计细节

### 三屏布局

`gogogo.1.sh` 在创建开发会话后，开发窗口调整为三 pane：

- pane 0：后端原始日志
- pane 1：前端原始日志
- pane 2：监控摘要屏

布局目标不是平均三等分，而是优先保证前后端日志有足够宽度，监控 pane 作为辅助摘要区域。第一阶段建议采用：

- 上方左右并排展示后端与前端日志
- 下方横向铺开监控 pane

这样可以同时满足两个要求：

- 左右两屏仍然能并排观察原始输出
- 第三屏在底部可展示完整的看板和滚动事件流，不容易被过窄宽度限制

### 工作区隔离的数据目录

为避免不同工作区、worktree 或并行开发会话互相污染，每个工作区使用独立的运行时目录。建议使用项目根目录下的专属临时目录，例如：

- `.tmp/gogogo/<session-suffix>/backend.log`
- `.tmp/gogogo/<session-suffix>/frontend.log`
- `.tmp/gogogo/<session-suffix>/monitor.state`
- `.tmp/gogogo/<session-suffix>/monitor.events.log`

其中 `<session-suffix>` 复用当前已有的工作区 session 后缀生成逻辑，确保和 `tmux` session 命名保持同源。

该目录用于开发态临时文件，不纳入长期持久化，也不作为业务数据目录。

### 前后端输出采集方式

后端与前端 pane 的启动命令保持原有核心命令不变：

- 后端核心命令仍然是 `air`
- 前端核心命令仍然是 `bun install && bun run dev`

但执行方式需要包装为“终端可见 + 同步落日志”。目标效果为：

- 开发者在 pane 中仍然看到完整原始输出
- 同一份输出被追加写入各自日志文件，供监控脚本读取

日志文件在每次新建开发会话时清空重建，避免第三个 pane 读取到上一次运行的陈旧状态。

### 监控脚本职责

新增独立监控脚本，例如 `gogogo.dev.monitor.sh`。它只负责读取运行时日志并生成人话摘要，不负责启动前后端进程。

该脚本持续运行在第三个 pane 中，职责包括：

- 初始化运行开始时间
- 读取前端与后端日志文件
- 根据匹配规则推断当前状态
- 渲染固定看板
- 记录并输出新的变化事件
- 聚合 warning 和 error 提示

该脚本不得直接依赖 `tmux capture-pane` 作为主要数据来源，也不应修改前后端业务进程。

### 第三屏展示形态

第三屏采用“固定看板 + 事件流”的双区结构。

固定看板区域用于持续刷新，始终展示当前全局状态，至少包括：

- 当前工作区 session 名
- 后端状态：未启动 / 启动中 / 已就绪 / 异常
- 前端状态：未启动 / 启动中 / 已就绪 / 异常
- 后端访问地址
- 前端访问地址
- 后端启动耗时
- 前端启动耗时
- 最近一次变化摘要
- warning 数量
- error 数量
- 日志最后更新时间

事件流区域用于只追加真正的新事件，示例包括：

- `后端开始输出，正在启动`
- `前端依赖安装完成，开发服务器启动中`
- `前端已就绪，地址 http://localhost:5173，耗时 8.1s`
- `后端已就绪，地址 http://localhost:3000，耗时 4.3s`
- `前端端口变化：5173 -> 5174`
- `检测到 warning：deprecated dependency optimize hint`
- `检测到后端 error：panic: database connection failed`

事件流只在状态发生变化时输出，避免每次轮询都重复刷同一句摘要。

### 地址识别规则

#### 前端地址

前端地址优先从 Vite 开发输出中提取。第一阶段匹配常见输出模式，例如：

- `Local:   http://localhost:5173/`
- `Local:   http://127.0.0.1:5173/`
- 其他含 `http://localhost:` 或 `http://127.0.0.1:` 的 Vite 本地地址行

一旦提取到新地址：

- 看板中的前端地址立即更新
- 若与旧地址不同，则事件流记录“地址变化”

#### 后端地址

后端地址优先从后端启动日志中提取监听地址或端口相关信息。第一阶段允许以下分层策略：

1. 若日志中能稳定匹配到监听地址，则标记为“已确认地址”
2. 若只能匹配到端口，则组装为 `http://localhost:<port>`
3. 若日志中没有稳定地址信息，则显示“待日志确认”，必要时可附带“默认推断地址”但必须明确标识为推断，不得伪装成已确认

这样可以避免第三屏把猜测地址当成事实展示。

### 启动耗时定义

前后端启动耗时都以 `gogogo.sh 1` 创建当前开发会话的时刻作为起点。

结束时刻定义如下：

- 前端：首次检测到有效本地访问地址时，视为前端已就绪
- 后端：首次检测到后端监听成功、服务启动完成或明确可推断 ready 的日志时，视为后端已就绪

若某一侧尚未就绪，则看板中显示“启动中”。

启动耗时一旦记录成功，应在本次会话内保持稳定，除非检测到明确的重启事件。若后续输出表明前端或后端重新启动，则应重新计算该侧耗时，并记录一条“重新启动”事件。

### 变化事件判定

第三屏需要关注的变化不只是“就绪”。第一阶段应至少识别以下事件类别：

- 首次开始输出
- 启动成功
- 启动失败
- 地址变化
- 后端重编译 / 重启
- 前端重新编译 / 重新优化依赖
- warning 新增
- error 新增

事件去重规则应尽量保守：

- 同一条日志内容连续重复出现时，只记录一次摘要
- 状态未变化时，不重复输出相同人话描述
- 若同类 warning 连续多次出现，可累计计数，但事件流不需要无限重复打印

### warning 与 error 提取

监控脚本需要对前后端日志进行基础分级提取。第一阶段以关键词规则为主，不做复杂语义理解。

建议分层如下：

- warning 关键词：`warn`、`warning`、`deprecated`
- error 关键词：`error`、`panic`、`fatal`、`failed`

需要注意两点：

- 关键词匹配应尽量大小写不敏感
- 某些工具会把普通说明行带上 `warning` 文案，因此摘要应尽量截取核心句子，而不是整段原始日志全部回放

第三屏初版不追求百分之百精准分类，但应保证“明显的 warning / error 能看见，且不会把所有普通日志都当告警”。

### 刷新与资源控制

固定看板采用周期刷新，建议默认每 1 秒刷新一次。

事件流不按轮询重复输出，而是只在检测到新事件时追加。

为控制资源占用：

- 监控脚本每次仅读取日志的新增部分或有限尾部内容
- 不在每次轮询时全量重扫整个大日志文件
- 不额外引入高频子进程风暴

### 清理与重启行为

开发会话启动前应清理当前工作区的旧进程与旧运行时目录，确保新会话使用干净日志。

开发会话通过 `gogogo.sh stop` 停止时，也应同步清理本工作区对应的开发进程与临时运行文件，避免残留日志误导下一次监控 pane。

若选择“关闭当前工作区会话并重启前后端”或“关闭所有开发会话并重启前后端”，也应保证第三屏使用的是当前新会话的全新日志和状态文件。

## 文件与职责建议

本次设计建议影响以下文件：

- `gogogo.1.sh`
  负责创建三 pane 布局，准备运行时目录，并让前端、后端、监控脚本分别进入对应 pane

- `gogogo.lib.sh`
  负责补充运行时目录、日志路径、状态文件路径等共享 helper，并扩展清理逻辑

- `gogogo.dev.monitor.sh`
  新增文件，负责解析日志、维护状态、渲染固定看板和输出事件流

- `tests/gogogo_dev_backend_command_test.sh`
  更新为兼容三 pane 结构，并继续验证后端核心命令仍为 `air`

- 新增或调整 `tests/gogogo_*` shell 测试
  验证第三个 pane 已接入、日志落盘已接入、监控脚本已接入、旧日志清理已接入

## 错误处理

第一阶段需要覆盖以下错误路径：

- 监控脚本启动时日志文件尚未产生，应显示“等待日志输出”，而不是直接报错退出
- 前端或后端长时间未出现 ready 信号，应保持“启动中”并可附带最近输出时间
- 若监控脚本读取日志失败，应在第三屏显示明确错误提示，而不是静默空白
- 若无法创建运行时目录或日志文件，本地开发启动应快速失败并提示具体路径

## 测试与验证

实现验证至少应包含以下内容：

- `bash -n` 检查新增和修改后的 shell 脚本语法
- 更新现有 shell 测试，确保它们不再假设只有两个 pane
- 新增 shell 测试验证三 pane 布局命令存在
- 新增 shell 测试验证前后端命令已接入日志落盘逻辑
- 新增 shell 测试验证监控脚本命令已在第三 pane 启动
- 手工运行 `./gogogo.sh 1`，确认可看到三屏布局
- 手工确认第三屏能识别前端地址、至少一种后端 ready 信号、warning/error 摘要与启动耗时

第一阶段不强求对监控脚本做完整端到端自动化测试，但至少要让脚本结构可被 shell 测试静态验证，核心解析函数可在后续迭代中继续收敛。

## 风险与处理

### 后端 ready 信号可能不稳定

不同运行路径下，后端启动日志格式可能不完全一致。第一阶段必须接受“后端地址和 ready 识别不如前端稳定”这一现实。

处理策略：

- 优先支持当前最常见日志样式
- 若日志不够明确，则在看板中显示“待日志确认”而不是伪造状态
- 将匹配规则设计为可独立补充，而不是写死在布局逻辑里

### 前端启动耗时包含依赖安装时间

当前前端命令是 `bun install && bun run dev`，因此前端 ready 耗时会自然包含安装阶段时间。

处理策略：

- 将这一时间定义为“前端开发环境可访问耗时”，而不是单纯 `vite` 启动耗时
- 第三屏事件流可以额外提示“依赖安装完成，正在启动前端服务”，让耗时来源更易理解

### 日志量可能较大

前后端输出长期运行后，日志文件可能变大。

处理策略：

- 会话级日志只服务于当前开发期
- 每次新会话启动时清空旧日志
- 监控脚本只读取新增部分或有限尾部内容，不做全量扫描

## 设计结论

本次设计将 `gogogo.sh 1` 从“前后端双屏并排日志”升级为“三屏开发监控工作台”：

- 前后端继续保留原始日志 pane
- 通过工作区独立日志把原始输出稳定落盘
- 第三屏通过独立监控脚本，将原始输出转成固定看板与人话事件流

这样既保留了现有开发习惯，也补上了“访问地址、启动速度、发生了什么变化、有哪些警告”这些需要开发者快速识别的信息。
