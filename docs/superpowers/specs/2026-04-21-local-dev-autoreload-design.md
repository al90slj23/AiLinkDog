# 本地开发自动重载设计

## 背景

当前项目的本地开发主入口是仓库根目录下的 `./gogogo.sh dev`。该入口会通过 `tmux` 分屏同时启动前后端，保持前后端日志并排可见，符合当前仓库的日常开发习惯。

从现有脚本与前端配置来看：

- 前端 `web/package.json` 的 `dev` 脚本是 `vite`
- 前端开发态天然具备 HMR / 自动刷新能力，多数页面与样式改动不需要手动重启
- 后端当前开发态仍以直接启动 Go 进程为主，本地修改 Go 源码后，通常需要重新编译并重启服务才能看到变化

本次讨论的目标不是引入生产级热更新能力，而是优化本地开发体验，让现有 `gogogo.sh + tmux` 工作流继续保留，同时把后端改为自动编译重启。

## 目标

本次设计聚焦一个明确目标：在不改变现有本地开发入口形态的前提下，优化前后端的反馈速度与操作成本。

具体目标如下：

- 保持 `./gogogo.sh dev` 作为本地开发唯一入口
- 保持 `tmux` 前后端并排显示日志的使用方式
- 前端继续沿用 `vite` 开发服务器，不引入第二套前端开发方案
- 后端在检测到相关源码变化后自动重新编译并重启
- 连续保存多个后端文件时，尽量合并成一次重启，减少打扰
- 后端编译失败时，旧服务继续运行，避免把本地环境直接打断

## 非目标

本次明确不做以下事情：

- 不尝试为 Go 后端实现真正意义上的运行时热替换
- 不改变生产环境启动、构建、部署或发布方式
- 不改造现有前端 HMR 机制
- 不为前端和后端分别引入新的独立开发命令入口
- 不顺带重构 `gogogo.sh` 的整体菜单结构与其他命令行为
- 不在第一阶段引入大量运行时动态配置刷新机制

## 现状判断

### 前端

前端目前使用 `vite` 作为开发服务器。对 React 组件、页面、样式和大部分前端模块改动，`vite` 会通过 HMR 或自动刷新将结果反馈到浏览器，因此前端开发态已经具备“尽量不手动重启”的能力。

但这并不意味着前端永远不需要重启。对于开发配置、环境变量、依赖安装结果等变更，前端开发服务器仍可能需要重新启动或手动刷新。这属于前端开发工具本身的正常边界，不在本次设计重点范围内。

### 后端

Go 后端与前端开发工具的机制不同。正在运行的 Go 进程不会自动感知源码变化，因此修改后端源码后，想看到新行为，必须发生一次“重新编译 + 启动新进程”。

这意味着：

- 后端不能像前端 HMR 一样只替换局部模块
- 后端本地开发的可优化方向，不是真热替换，而是让“重新编译 + 重启进程”自动化，并把体验做得尽量轻量

## 方案选项

### 方案 A：保留手动重启，仅优化启动速度

继续沿用当前直接启动后端的方式，只通过减少初始化成本、缩短编译耗时来改善体验。

优点：

- 改动最小
- 不新增本地开发依赖

缺点：

- 不能解决“保存后自动反馈”的核心需求
- 开发者仍需频繁手动重启

### 方案 B：引入 `air` 作为后端 watcher

在本地开发环境中使用 `air` 监听后端源码变化。检测到变更后，由 `air` 自动重新编译并在成功后重启后端进程。

优点：

- 贴合 Go 项目的常见开发方式
- 与当前 `gogogo.sh + tmux` 工作流兼容度高
- 能把手动重启转换为自动反馈
- 配置能力完整，适合控制监听范围、防抖时间和忽略目录

缺点：

- 仍然不是运行时热替换，而是自动编译重启
- 连续保存时若配置不当，可能产生较频繁的编译输出

### 方案 C：使用 `CompileDaemon`、`reflex` 等其他 watcher

以其他文件监听工具替代 `air`，实现类似的自动重启效果。

优点：

- 也能满足“文件变更后自动重启”的基本需求

缺点：

- 与 `air` 相比，社区通用经验、现成配置和协作认知更弱一些
- 在本项目场景下，没有明显超过 `air` 的收益

## 方案选择

选择方案 B，即：保留现有前端 `vite` 开发方式，并为后端引入 `air`。

理由如下：

- 能最大程度保持现有 `./gogogo.sh dev` 的使用习惯不变
- 对仓库的侵入范围最小，只影响本地开发链路
- 能直接解决“后端改动后必须手动重启”的主要痛点
- `air` 足够成熟，便于长期维护与团队协作

## 设计细节

### 入口与工作流

保留现有本地开发入口：

- 开发者继续通过 `./gogogo.sh dev` 启动本地环境
- `tmux` 分屏结构继续保留
- 前端 pane 继续运行 `bun install && bun run dev`
- 后端 pane 从直接启动 Go 进程切换为运行 `air`

因此，整体工作流不变，变化只体现在后端 pane 的启动命令与输出内容上。

### 后端自动重载的真实语义

本次设计需要明确区分“前端热更新”和“后端自动重启”：

- 前端通过 `vite` 优先尝试 HMR 或自动刷新，尽量避免完整重启
- 后端通过 watcher 发现文件变化后重新编译，再由新进程接管服务

因此，本次后端优化的准确表述应为“自动编译重启”，而不是真正的运行时热替换。

### 监听范围

第一阶段采用保守策略，默认只监听后端相关 `.go` 文件。

应纳入监听的范围包括但不限于：

- 仓库根目录下的 `main.go`
- 后端源码目录，例如 `controller/`、`service/`、`model/`、`router/`、`relay/`、`common/`、`dto/`、`types/`、`middleware/`、`setting/`

应明确排除的范围包括：

- `web/`
- `.git/`
- `node_modules/`
- `tmp/`
- 日志目录
- 前端构建产物目录
- 文档目录，如 `docs/`、`.ai/`
- IDE 临时文件与其他无关生成文件

选择该策略的原因是：先优先解决最常见的 Go 源码修改反馈问题，避免无关文件变化导致后端频繁重启。

### 配置文件监听策略

对于 `.env`、YAML、JSON、TOML 等配置文件，第一阶段不默认纳入自动重启监听。

原因如下：

- 并非所有配置文件变化都应触发后端重启
- 不同配置的读取时机不同，有些是启动时加载，有些本来就不该由 watcher 处理
- 一次性把全部配置文件纳入监听，容易带来噪声和误触发

后续如果确认某类配置文件在本地开发中频繁变更，且其变化本就要求服务重启，再单独补入监听名单。

### 连续修改时的防抖策略

为减少持续修改多个后端文件时的频繁重启，watcher 需要启用短暂的防抖或延迟触发机制。

其行为预期为：

- 检测到变化后，不立刻开始编译
- 先等待一个较短缓冲时间
- 若缓冲时间内仍有新的保存动作，则合并为同一次编译与重启

这样可以把短时间内连续保存多个文件的动作尽量折叠成一次自动重启，避免本地开发时输出过于频繁。

### 编译失败时的行为

后端 watcher 必须保证以下体验边界：

- 新代码编译失败时，后端 pane 明确输出错误信息
- 编译失败不会替换当前已运行的旧进程
- 旧服务继续可用，直到下一次编译成功

这是本次设计中的关键体验要求。否则开发者在保存一个尚未完成的中间态文件时，整个本地服务会被打断，影响调试效率。

### 日志与可见性

自动重载相关日志应直接出现在 `tmux` 的后端 pane 中。

开发者应能从后端 pane 直接观察到以下状态：

1. 检测到文件变更
2. 开始重新编译
3. 编译成功并重启服务
4. 编译失败但旧服务继续运行

这样可以保持当前“前后端日志并排观察”的工作方式，不需要新增其他可视化工具。

## 影响范围

本次设计的实现范围应控制在本地开发辅助链路：

- 新增一个后端 watcher 配置文件，例如 `.air.toml`
- 修改 `gogogo` 的本地开发子脚本，使后端 pane 运行 `air`
- 视实际需要微调 `gogogo.lib.sh` 中与进程清理相关的匹配逻辑
- 补充本地开发说明文档，明确前端与后端的更新机制差异

本次不应触碰以下范围：

- 生产启动脚本
- 发布与部署脚本
- Go 应用业务逻辑
- 前端业务代码和现有 `vite` 配置主路径

## 风险与处理

### 风险 1：连续保存时频繁自动编译，影响开发节奏

处理方式：

- 使用防抖 / 延迟触发
- 严格限制监听范围
- 排除前端、文档、日志和临时目录

### 风险 2：编译失败导致本地服务被打断

处理方式：

- 确保 watcher 在编译失败时保留旧进程
- 在日志中清晰区分“编译失败”和“服务已重启”

### 风险 3：`gogogo.sh` 的 stop / cleanup 逻辑与实际后端进程不一致

处理方式：

- 让开发脚本中的进程匹配模式与 `air` 的实际运行命令保持一致
- 在实现阶段补充一次启动与停止链路验证

## 验证方式

实现后需要至少验证以下场景：

1. 执行 `./gogogo.sh dev`，确认 `tmux` 分屏仍正常工作
2. 确认前端 pane 继续按原方式启动 `vite`
3. 修改任意后端 `.go` 文件，确认后端 pane 自动触发重新编译与重启
4. 连续快速保存多个后端 `.go` 文件，确认通常只触发一次合并后的重启
5. 人为制造一个后端编译错误，确认旧服务继续运行，后端 pane 仅输出编译错误
6. 执行 `./gogogo.sh stop`，确认相关本地开发进程能被正确清理

## 结论

本次设计采用“前端继续 `vite`、后端引入 `air` 自动编译重启、统一保留 `gogogo.sh + tmux` 工作流”的最小改动方案。

它不追求 Go 运行时热替换，而是把后端本就必须发生的“重新编译 + 重启”过程自动化、可视化，并通过监听范围控制、防抖和编译失败保留旧进程等手段，把本地开发体验优化到更稳定、顺手的状态。
