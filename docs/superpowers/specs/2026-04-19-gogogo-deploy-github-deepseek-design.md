# gogogo 部署与 GitHub 提交流程设计

## 摘要

为当前 `AiLinkDog` 仓库的 `gogogo.sh` 增加一个一级主菜单项 `部署`，用于承接后续部署相关操作。

本次先实现其中的 `仅上传到 GitHub 仓库` 流程，并在该流程中接入 DeepSeek 生成中文提交摘要、人工确认与编辑、提交到 `al90slj23/ailinkdog` 主分支，以及可选地向 `QuantumNous/new-api` 创建 Pull Request。

这次设计只落地 GitHub 相关流程，不提前实现服务器部署逻辑，但会为未来的 `上传到服务器并同时上传到 GitHub` 与 `仅上传到服务器` 预留稳定的菜单入口。

## 目标

- 在 `gogogo.sh` 中新增一级主菜单项 `部署`。
- 在 `部署` 菜单下提供 3 个二级选项，先真实实现 `仅上传到 GitHub 仓库`。
- 复用参考仓库中“AI 生成摘要后人工确认再继续”的交互思路，但针对当前仓库重写更合适的中文提示词与执行流程。
- 使用环境变量 `DEEPSEEK_APIKEY_GOGOGOSH_GITHUB` 调用 DeepSeek。
- 生成中文提交标题与中文提交正文，并允许人工确认、重生成、手动修改或完全自定义。
- 将最终确认结果用于 `git commit`：第一行作为标题，正文作为 body。
- 默认将变更推送到 `al90slj23/ailinkdog` 的 `main` 分支。
- 推送成功后，支持可选地向 `QuantumNous/new-api` 创建 PR，且自动探测目标仓库默认分支。

## 非目标

- 本次不实现真实的服务器上传、rsync、SSH 运维或远端部署动作。
- 本次不自动修改 Git 远端配置，不重写 `origin` 指向。
- 本次不强制创建 PR；PR 仅作为 GitHub 提交成功后的可选后续步骤。
- 本次不直接照搬参考仓库的 AI 提示词，因为参考仓库当前实现主要是占位回退，不适合直接复用。
- 本次不把未经确认的 AI 输出直接用于提交或 PR。

## 背景与参考

参考仓库 `YYSYYF` 中可以复用的是整体交互结构，而不是完整的 AI 能力实现：

- `gogogo.git.sh` 中的“先生成摘要，再人工确认，再提交”的交互思路可以借鉴。
- `gogogo.ai.sh` 中保留了 AI 调用接口形态，但当前实现主要是本地回退，并没有提供一套适合当前仓库的成熟 DeepSeek 中文提示词。

因此，当前仓库应保留其流程设计优势，同时重写以下内容：

- 中文提示词
- DeepSeek API 调用实现
- 面向 `AiLinkDog` 的提交与 PR 流程
- 与当前 `gogogo.sh` 架构匹配的菜单与脚本拆分

## 菜单结构

### 一级菜单

在当前 `gogogo.sh` 主菜单中新增一级菜单项：

- `6. 部署`

同时支持命令别名，例如：

- `./gogogo.sh 6`
- `./gogogo.sh deploy`

保留当前已有的 `dev/build/test/lint/clean` 等现有入口，不改变其既有行为。

### 二级菜单

`部署` 菜单由新脚本负责展示，包含：

1. 上传到服务器并同时上传到 GitHub
2. 仅上传到 GitHub 仓库
3. 仅上传到服务器
0. 返回

其中：

- 本次只真正实现第 `2` 项。
- 第 `1`、`3` 项先输出明确提示，例如“暂未实现，后续安排”，不执行任何假动作。

## 文件结构

本次新增或调整的脚本边界如下：

- `gogogo.sh`
  - 负责新增 `部署` 一级入口。
  - 负责别名归一化与一级调度。
- `gogogo.6.sh`
  - 负责 `部署` 二级菜单。
  - 负责调用“仅上传到 GitHub 仓库”的实际执行函数。
- `gogogo.git.sh`
  - 负责 GitHub 提交流程。
  - 负责 git 状态检查、提交、推送、可选 PR 创建。
- `gogogo.ai.sh`
  - 负责 DeepSeek 调用与本地回退。
  - 负责生成提交标题与提交正文。
- `gogogo.lib.sh`
  - 保持为轻量公共库。
  - 仅补充少量通用工具，不承载复杂业务流程。

这样拆分的原因是：

- `部署` 二级菜单未来还要承接服务器部署，不应塞进主入口。
- AI 调用和 Git 操作职责不同，分文件更容易维护。
- 保持当前 `gogogo.*.sh` 的心智模型，降低未来扩展成本。

## GitHub 提交流程

### 总体流程

当用户进入 `部署 -> 仅上传到 GitHub 仓库` 后，流程如下：

1. 检查前提条件。
2. 收集本次变更信息。
3. 调用 DeepSeek 生成中文提交标题与中文提交正文。
4. 进入人工确认环节。
5. 使用最终确认内容执行 `git commit`。
6. 将提交推送到 `al90slj23/ailinkdog` 的 `main` 分支。
7. 推送成功后，询问是否额外向 `QuantumNous/new-api` 创建 PR。

### 前提检查

执行前应检查：

- 当前在仓库根目录。
- `git` 命令可用。
- 已存在 Git 变更；如果没有任何变更，应直接提示并结束。
- 若需要调用 DeepSeek，环境变量 `DEEPSEEK_APIKEY_GOGOGOSH_GITHUB` 已设置。
- 若用户后续选择创建 PR，则要求 `gh` 命令可用。

默认不主动修改 Git 远端配置。该流程假定当前仓库本身已经正确配置为 `al90slj23/ailinkdog`。

### 变更收集

用于摘要生成的上下文至少应包含：

- `git status --short`
- `git diff --cached`
- 当暂存区为空时，补充 `git diff`

必要时还可以追加：

- 当前分支名
- 最近 3 到 5 条提交信息，用于对齐本仓库的提交风格

但输入需要控制长度，避免将过大 diff 全量发送给 DeepSeek。设计上应优先发送：

- 文件列表
- diff 摘要
- 截断后的关键上下文

## DeepSeek 摘要生成设计

### 输入与输出

DeepSeek 的任务是生成两部分内容：

- `title`：用于 `git commit` 第一行的中文标题
- `body`：用于 `git commit` 正文的中文说明

输出格式应固定，便于 shell 解析，例如：

```text
TITLE: 修复 xx 问题并补充 xx 流程
BODY:
本次修改主要做了以下调整：
1. ...
2. ...

这样修改的原因是：
1. ...
2. ...
```

### 提示词要求

提示词应使用中文，并明确要求：

- 必须基于真实变更生成，不得编造未发生的修改。
- 风格简洁、真实、可复核。
- 标题不要过长，适合 `git commit` 第一行。
- 正文应说明“改了什么”和“为什么这样改”。
- 不要写空泛套话，不要夸大结果。
- 不要编造验证结果。
- 输出使用中文。

### DeepSeek API Key

固定读取：

- `DEEPSEEK_APIKEY_GOGOGOSH_GITHUB`

若该变量未设置，应明确报错并终止 AI 生成功能，不静默降级为其他环境变量。

### 失败回退

若 DeepSeek 调用失败、超时、响应格式不合法，系统应回退到本地规则生成一组基础可用结果：

- `title`：例如 `更新项目文件`、`调整 gogogo 部署脚本` 之类的中文短标题
- `body`：根据变更文件范围生成简短中文正文

本地回退结果仍必须进入人工确认环节，不允许自动直接提交。

## 人工确认与编辑流程

AI 结果生成后，必须进入人工确认环节。用户可选择：

1. 接受当前摘要
2. 重新生成 AI 摘要
3. 编辑当前摘要
4. 完全自定义输入
0. 取消本次操作

### 编辑当前摘要

“编辑当前摘要”表示以 AI 当前生成结果为底稿：

- 先展示当前 `title`
- 再展示当前 `body`
- 分别允许用户修改标题与正文

### 完全自定义输入

“完全自定义输入”表示不依赖当前 AI 结果，直接从空白输入：

- 手动输入 `title`
- 手动输入 `body`

### 提交流程约束

- 未经确认的 AI 输出不能直接进入 `git commit`。
- 最终确认版本才允许执行提交。
- 如果用户选择取消，则整个流程直接结束，不产生任何 Git 写操作。

## Git 提交与推送策略

### 提交策略

最终确认后的摘要用于 `git commit`：

- 第一行使用最终 `title`
- 正文使用最终 `body`

执行方式应采用显式 title + body 的方式，而不是将所有内容塞成一行。

### 暂存策略

本次流程将沿用参考仓库“统一暂存再提交”的模式，即在提交前执行：

- `git add .`

设计上默认用户是通过该入口做一次完整上传动作，因此采用全量暂存模型更贴近当前用户习惯。

### 推送策略

推送行为按当前仓库的实际使用习惯处理：

- 默认推送到主分支 `main`
- 不使用“当前分支自动推送”的通用协作模式作为默认行为
- 不主动修改远端 URL

如果当前不在 `main` 分支：

- 先明确提示当前分支不是 `main`
- 让用户确认是否继续
- 若用户拒绝，则停止流程

这样可以满足“默认直接提交到主分支”的使用习惯，同时避免误把其他分支内容直接推上去。

## PR 创建策略

### 触发时机

PR 创建不是默认动作，而是在 GitHub push 成功后追加询问：

- 是否同时向 `QuantumNous/new-api` 创建 PR

### 目标仓库与分支

- 目标仓库固定为 `QuantumNous/new-api`
- base 分支自动探测目标仓库默认分支
- head 使用当前仓库当前分支

### PR 标题与正文

PR 标题优先复用最终确认后的 `title`。

PR 正文不应直接原样粘贴 AI 输出，而应基于最终确认摘要整理为更适合 GitHub 的结构化正文，例如：

```markdown
## Summary

- ...
- ...

## Why

- ...

## Verification

- Not run locally
```

其中：

- `Summary` 与 `Why` 基于最终确认后的中文摘要整理。
- `Verification` 不得编造结果；若本次未运行验证，则必须如实写明。

该设计与仓库现有规范保持一致：PR 摘要应由提交者基于真实理解整理，不应直接粘贴未经处理的 AI 输出。

## 交互与提示风格

所有用户可见提示默认使用中文，风格要求：

- 简洁直接
- 明确指出当前步骤
- 明确指出下一步会发生什么
- 对失败原因给出具体信息

例如：

- 缺少 API Key
- 当前没有可提交的变更
- 当前分支不是 `main`
- `gh` 未安装，无法创建 PR

## 错误处理

应覆盖以下失败场景：

- 缺少 `git`
- 缺少 `gh`
- 缺少 `DEEPSEEK_APIKEY_GOGOGOSH_GITHUB`
- DeepSeek 接口失败或输出格式非法
- 没有变更可提交
- `git commit` 失败
- `git push` 失败
- PR 创建失败

处理原则：

- 优先给出明确错误原因。
- 除 DeepSeek 失败允许回退外，其余关键失败应直接终止，不做隐式继续。
- 不执行任何破坏性 Git 操作。

## 验证范围

实现完成后的验证至少包括：

- `bash -n gogogo.sh gogogo.lib.sh gogogo.6.sh gogogo.git.sh gogogo.ai.sh`
- 验证主菜单与 `deploy` 二级菜单显示正常
- 验证未实现的第 `1`、`3` 项只做占位提示
- 验证缺少 API Key 时的错误输出
- 验证 DeepSeek 回退逻辑可用
- 验证确认、重生成、编辑、自定义、取消等交互分支
- 验证 `main` 分支与非 `main` 分支提示逻辑
- 验证 push 成功后可选创建 PR 的流程

如果由于本地环境限制无法完整执行 PR 创建，应至少验证：

- `gh` 依赖检测逻辑正确
- base 分支探测命令与 PR 创建命令构造正确

## 设计边界

本次设计严格限定在 `gogogo.sh` 的部署与 GitHub 提交流程，不修改应用本身的 Go 或前端业务逻辑，不修改受保护的 `new-api` / `QuantumNous` 标识，也不引入与当前目标无关的自动化功能。

未来若要补齐服务器上传能力，应复用本次新增的 `部署` 一级入口与二级菜单框架，在不破坏已有 GitHub 提交流程的前提下渐进扩展。
