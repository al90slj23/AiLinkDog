# AI 记忆体系设计

## 概述

为当前 `AiLinkDog` 仓库建立一个项目内的 `./ai` 六层记忆体系。新体系保留旧项目的分层组织方式，但所有内容都必须基于当前仓库的真实状态重写，而不是直接复制旧项目内容。

这套记忆体系的目标，是成为未来 AI 与人类协作时的长期主记忆。它应当准确反映当前代码、当前文档、当前工具链，以及当前仓库里实际生效的工程规则。

## 目标

- 在当前项目中建立一个 `./ai` 六层记忆体系。
- 保留旧项目的分层结构与命名风格。
- 将内容重写为符合当前仓库真实状态的版本，而不是沿用旧项目假设。
- 让这套体系成为长期可用的主记忆，而不是一次性的初始化脚手架。
- 把“全程使用中文沟通交流”确立为当前项目的第一条额外开发规范。
- 把“项目 specs 默认使用中文编写，方便用户直接阅读与维护”确立为正式规则。

## 非目标

- 不机械复制旧项目 `.ai` 内容。
- 不继承与当前仓库冲突的旧规则。
- 不把旧项目的架构、前端栈或 UI 规范当作当前仓库的权威来源。
- 本阶段不建设自动知识图谱引擎、自动同步平台或自动摘要流水线。

## 来源优先级

新的 `./ai` 内容必须按照以下优先级建立：

1. 当前仓库的代码与目录结构。
2. 当前仓库现有规范与文档，尤其是 `AGENTS.md`、`CLAUDE.md` 与 `README*`。
3. 当前会话里已经形成的设计与计划文档（如有相关内容）。
4. 旧项目的 `ZERO` 与 `.ai`，但仅用于借鉴结构与层级职责。

如果旧 `.ai` 内容与当前仓库冲突，必须以当前仓库为准。

## 分层模型

新体系采用六层结构，沿用旧项目的分层命名风格。

### `./ai/L0#Execution/`

定位：执行层，用于放工作流、计划、规格、模板等动态执行内容。

初始范围：

- `README.md`
- `specs/`、`plans/`、`hooks/`、`templates/`、`workflows/` 的目录骨架
- 清楚说明本层存放执行工件，而不是项目核心真相

这一层应保持偏执行、偏动态、偏轻量。

### `./ai/L1#Overview/`

定位：AI 和人类首次阅读的概览层。

初始范围：

- `README.md`
- `guide.md` 作为主入口文档

内容要求：

- 项目定位与用途
- 核心技术栈
- 后端分层架构
- 前端与 i18n 配置
- 关键目录结构
- 最重要的开发规则
- 常用命令
- 额外第一条规则：全程使用中文沟通交流
- 额外第二条规则：spec 默认使用中文编写

这一层应简洁、稳定、可首读。

### `./ai/L2#Index/`

定位：结构化导航与记忆索引层。

初始范围：

- `README.md`
- `product.md`
- `tech.md`
- `structure.md`
- `conventions.md`
- `toc.md`

这一层的职责是建立项目知识地图，告诉未来 AI 应该去哪里找产品信息、技术约束、目录结构与开发规范，而不是重复整套标准正文。

### `./ai/L3#Standards/`

定位：长期规范与详细知识层。

初始范围：

- `README.md`
- `standards/README.md`
- 围绕当前仓库真实情况组织的核心规范：
  - 元规范与使用方式
  - 架构与分层
  - 后端规范
  - 前端规范
  - 质量规范
  - 工具规范
  - AI 记忆体系规范

预期主题包括：

- 架构分层与仓库拆解方式
- `common/json.go` 使用规则
- SQLite / MySQL / PostgreSQL 三兼容规则
- Bun 优先的前端工作流
- relay / channel 集成约束
- DTO 显式零值的指针语义
- i18n 规范
- 变更边界与验证纪律
- `gogogo.sh` 使用方式
- AI 记忆体系维护规则
- 中文沟通与中文 spec 要求

这一层是长期主规则正文层。

### `./ai/L4#Changelog/`

定位：历史决策与操作日志层。

初始范围：

- `README.md`
- 一篇初始化日志，至少记录：
  - 为什么要建立新的 `./ai`
  - 初始化基于哪些来源
  - 哪些旧规则被明确认定为不兼容并舍弃
  - 六层结构如何适配到当前仓库

这一层应能让未来变更可追溯。

### `./ai/L5#Knowledge/`

定位：关系导向的知识层。

初始范围：

- `README.md`
- 一篇或多篇种子知识文档，用于描述当前项目中的核心关系，例如：
  - Router -> Controller -> Service -> Model
  - relay 请求/响应转换路径
  - 前后端 i18n 的分离关系
  - 前端构建产物与后端 embed 的关系
  - Bun、Vite、Go、Docker、`gogogo.sh` 的工具关系

这一层应强调关系与系统地图，不重复 L3 的规范正文。

## 命名与书写风格

- 保留旧项目的层级命名风格，例如 `L0#Execution`、`L1#Overview`。
- 新记忆内容统一使用中文书写。
- 未来项目 specs 默认使用中文书写。
- 文件名保持稳定、明确、可读。
- 优先使用高信息密度、低噪音的文档，而不是大量占位空文档。
- 当旧术语已经不适用于当前项目时，不应继续沿用。

## 当前项目真实情况必须覆盖旧记忆

设计必须明确尊重以下当前仓库事实：

- 当前前端真实使用的是 Semi Design UI，而不是旧项目假设的 Ant Design。
- 当前前端包管理器优先使用 Bun。
- 当前后端是一个包含大量 relay 适配器的 Go AI gateway / proxy。
- 当前项目必须兼容三种数据库。
- 业务代码中的 JSON 操作必须使用 `common/json.go` 包装。
- `new-api` 与 `QuantumNous` 相关标识属于受保护信息，不得移除或改写。
- relay 请求中的可选标量字段必须通过指针类型保留显式零值语义。
- 当前项目新增正式规则：全程使用中文沟通交流。
- 当前项目新增正式规则：spec 默认使用中文编写。

## 预期文件骨架

第一版实现必须产出一套可直接使用的六层结构，而不是只有空目录。

最小结构如下：

## 信息设计策略

整套体系应保持清晰的层次分工：

- L1 回答：这个项目是什么，最重要的规则是什么？
- L2 回答：我要去哪里找产品、技术、结构与规范？
- L3 回答：当前项目有哪些长期有效的详细规则？
- L4 回答：这些内容为什么会这样建立、何时发生变化？
- L5 回答：系统中关键概念和子系统之间如何关联？

这种分工可以减少重复，并让记忆体系在长期维护中保持稳定。

## 验证要求

实现完成后应验证：

- `./ai` 六层结构完整存在。
- L1、L2 与核心 L3 文档都已填充为仓库特定内容。
- 没有把旧项目冲突规则直接带入当前仓库。
- 中文沟通规则被明确写入。
- 中文 spec 规则被明确写入。
- 记忆文件与 `AGENTS.md`、`CLAUDE.md`、当前代码结构保持一致。

## 范围边界

本设计只覆盖当前仓库本地六层记忆体系的初始化与首版落地，不包括外部 IDE 平台自动同步、自动摘要流水线或高级知识图谱生成。这些能力可以在本地记忆体系稳定之后再逐步建设。
