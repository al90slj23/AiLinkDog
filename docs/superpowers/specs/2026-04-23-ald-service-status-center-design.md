# ALD 服务状态中心设计

## 背景

AiLinkDog 当前已经具备 AI 网关与后台管理能力，但缺少一套同时覆盖管理员完整视图与对外公开简洁展示的统一状态能力。

现有目标不是单独做一个纯监控工具，也不是将公开状态页当作后台的只读副本，而是建设一套统一的数据与能力域：

- 后台查看完整状态、明细、事件、配置与公开规则
- 公开页面向访客展示简洁、稳定、可快速理解的状态信息
- 公开展示风格可参考 `relay-pulse`，但产品边界、数据结构与后台能力保持 AiLinkDog 自主设计

## 目标

建设一套统一命名、统一数据口径、双视图输出的状态能力：

- 管理员侧提供完整版本
- 公开侧提供简洁版本
- 两端共享同一套底层状态聚合与判定逻辑
- 公开页不暴露后台敏感信息

## 非目标

- 不以 `relay-pulse` 为项目底座直接 fork 后改造
- 不将公开页做成后台完整数据的简单只读镜像
- 不在本阶段扩展为独立的多前端多仓库体系

## 正式命名

### 统一能力域名称

- 中文：`ALD 服务状态中心`
- 英文：`ALD Service Status Center`

这是该能力域在产品、文档、设计和后续实现中的正式名称。

### 后台入口命名

- 所属位置：控制中心侧边栏 `管理员` 分组下
- 菜单中文名：`服务状态中心`
- 菜单英文名：`Service Status Center`

### 公开入口命名

- 所属位置：全局顶部导航栏
- 导航文案中文：`服务状态`
- 导航文案英文：`Service Status`

### 公开页标题命名

- 中文：`ALD 服务状态`
- 英文：`ALD Service Status`

## 产品定位

`ALD 服务状态中心` 是 AiLinkDog 内部的一项平台能力，不是单独的外挂状态站，也不是纯粹的运维监控页。

其职责是：

- 对内承载完整状态分析、事件查看、公开规则配置与状态运营
- 对外承载统一、简洁、可信的状态展示

因此它的正确形态是：

- 同一后端能力域
- 两种信息暴露层
- 同一指标口径
- 不同展示深度

## 架构建议

推荐采用“同后端双视图”方案。

### 后端

后端维持单一状态能力域，负责：

- 原始状态采集结果接入
- 状态聚合与判定
- 事件生成与查询
- 后台完整接口输出
- 公开页脱敏接口输出

### 前端

前端在同一产品内拆为两套视图：

- 后台视图：管理员使用，完整数据
- 公开视图：访客使用，简洁数据

### 原则

- 后台与公开页共享同一底层数据口径
- 公开页不得直接消费后台接口
- 公开接口必须明确做展示裁剪与脱敏
- 命名统一使用 `status` 体系，避免与 `monitor`、`health` 混用

## 路由约定

### 后台路由

- `/admin/status`
- `/admin/status/overview`
- `/admin/status/services`
- `/admin/status/events`
- `/admin/status/settings`
- `/admin/status/public`

### 公开路由

- `/status`

后续如有独立公开域名需求，可平移到单独子域名，但当前设计不要求拆成独立站点。

## API 边界约定

### 后台接口前缀

- `/api/admin/status/...`

用于输出完整数据，包括但不限于：

- 服务状态明细
- 时间维度趋势
- 事件记录
- 配置与阈值
- 公开页配置项

### 公开接口前缀

- `/api/public/status/...`

用于输出经过裁剪后的公开数据，包括但不限于：

- 当前总体状态
- 公开服务列表
- 24h / 7d 简要趋势
- 最近公开事件
- 公告与维护信息

## 后台信息架构

后台首期建议拆成五个子页面。

### 1. 状态总览

用于快速了解整体健康度，建议包含：

- 全局状态摘要
- 当前异常服务数
- 24h / 7d 可用率摘要
- 延迟趋势
- 最近事件摘要

### 2. 服务状态

用于查看完整状态对象列表与当前表现，建议包含：

- 服务名称
- 状态
- 可用率
- 平均延迟
- 最近异常时间
- 分组/标签
- 是否公开展示

### 3. 事件记录

用于查看状态变化生命周期，建议包含：

- 故障事件
- 恢复事件
- 降级事件
- 维护窗口
- 公告记录

### 4. 公开页配置

用于定义公开页展示规则，建议包含：

- 公开服务开关
- 排序
- 显示名称
- 分组
- 公告文案
- 趋势显示开关
- 事件显示开关
- 页脚说明

### 5. 监控设置

用于管理内部状态判定与运行规则，建议包含：

- 探测频率
- 阈值设置
- 状态判定规则
- 告警规则
- 数据保留策略

## 公开页信息架构

公开页应坚持“简洁可读”而不是“后台精简版”的思路。

### 1. 顶部状态摘要

建议展示：

- 当前总体状态
- 最后更新时间
- 公告或维护提示

### 2. 服务状态列表

建议展示：

- 服务名称
- 当前状态
- 24h 可用率
- 平均延迟
- 最近更新时间

### 3. 历史趋势

建议支持：

- `24h`
- `7d`

只展示最必要的趋势，不承担后台级分析职责。

### 4. 最近事件

建议展示：

- 最近故障
- 最近恢复
- 维护通知

### 5. 页脚说明

建议展示：

- 数据口径说明
- 刷新时间说明
- 联系方式或帮助链接

## 信息暴露原则

后台可见但公开页默认不应暴露的信息包括：

- 内部请求日志
- 敏感供应商细节
- 渠道与密钥配置
- 完整模型映射细节
- 排障级别的内部指标

公开页应优先传达：

- 当前是否可用
- 哪些服务受影响
- 最近是否发生故障
- 用户是否需要等待或切换

## 对 `relay-pulse` 的参考边界

### 应参考的部分

- 公开页整体视觉节奏
- 状态列表或矩阵的信息层级
- 历史趋势展示方式
- 一眼可读的状态表达方式

### 不应直接照搬的部分

- 项目总体边界
- 后台结构
- 数据模型
- 配置体系
- 目录结构与技术实现细节

AiLinkDog 应借鉴的是 `relay-pulse` 的公开展示思路，而不是把自身平台能力建立在其项目结构之上。

## 方案选择结论

在以下三个方向中：

1. 直接 fork `relay-pulse` 并叠加后台
2. 完全从零重做全部形态
3. 以 AiLinkDog 现有平台为主，新增 `ALD 服务状态中心` 能力域，并让公开页参考 `relay-pulse`

本设计选择第 3 种。

原因如下：

- 更符合 AiLinkDog 现有平台主干
- 后台完整能力能自然延续当前系统结构
- 公开页可以吸收成熟展示经验
- 能避免将“监控工具产品边界”错误地变成主系统边界

## 首阶段实现范围建议

首阶段只需要把最小闭环建立起来：

- 确立命名与路由
- 落下后台菜单入口
- 落下公开页入口
- 提供后台总览、服务状态、事件记录三个核心页
- 提供公开页首页与公开状态接口
- 建立公开页配置能力，控制哪些服务可对外展示

监控设置、告警细化、更多趋势分析可以在后续迭代中逐步补全。

## 验收标准

当以下条件成立时，可认为首阶段方向正确：

- 后台管理员能从侧边栏进入 `服务状态中心`
- 公开访客能从顶部导航进入 `服务状态`
- 后台与公开页使用统一状态口径
- 公开页只展示允许公开的数据
- 公开页整体风格达到“参考 `relay-pulse`，但不是复制品”的目标
