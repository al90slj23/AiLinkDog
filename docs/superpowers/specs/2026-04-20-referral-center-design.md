# 邀请返利独立页面设计

## 背景

当前项目已经存在基础邀请能力，包括邀请链接、邀请人数、邀请奖励额度、邀请额度划转，以及后台中的邀请奖励额度设置。但这些能力目前主要散落在钱包管理和系统设置中，整体形态仍然偏向轻量级“邀请奖励”能力，尚未形成独立的“邀请返利业务中心”。

参考 `AiLinkDog-old` 后，确认旧项目已经演进出两块独立能力：

1. 面向普通用户的邀请返利中心。
2. 面向管理员的邀请返利管理页。

本次工作不是简单沿用旧项目路由和页面结构，而是在当前项目中按现有惯例新增两个独立页面，并将旧钱包中的邀请奖励块保留为迁移过渡入口，待新页面稳定后再移除。

## 目标

本次设计目标如下：

1. 新增普通用户可访问的独立页面 `邀请返利中心`。
2. 新增管理员可访问的独立页面 `邀请返利管理`。
3. 采用当前项目风格的平级控制台路由，而不是复用旧项目 `/console/user/*`、`/console/admin/*` 路径。
4. 用户首次创建邀请链接前，必须为当前账号选择并锁定返利方案。
5. 多邀请链接用于不同分发平台的投放与统计归因，而不是用于同账号下不同返利模式并存。
6. 保留钱包管理中的旧邀请奖励块，作为迁移期兼容入口；待新页面测试稳定后再移除。

## 不在本次范围内

本次明确不做以下增强项：

1. 不支持同一账号下不同邀请链接选择不同返利模式。
2. 不支持邀请链接覆盖账号已锁定的返利方案。
3. 不支持超过二级的返利层级扩展。
4. 不实现复杂自动审批规则引擎。
5. 不实现链接级独立结算规则。
6. 不在本期直接删除钱包管理中的旧邀请奖励块。

## 页面与路由

本次新增两个独立页面，均直接挂在 `/console/*` 平级路由下。

### 普通用户页面

- 中文名称：`邀请返利中心`
- 路由：`/console/referralcenter`

该页面面向普通登录用户，作为邀请返利业务的统一主入口。

### 管理员页面

- 中文名称：`邀请返利管理`
- 路由：`/console/referralmanage`

该页面面向管理员，统一承载返利配置、方案管理、提现审核与平台统计。

## 总体迁移策略

本次迁移采用“新增主入口，旧入口过渡保留”的方式推进。

1. 新增 `邀请返利中心` 与 `邀请返利管理` 两个独立页面。
2. 钱包管理中的旧邀请奖励块继续保留，不在本期直接删除。
3. 新页面完成联调与测试后，再移除钱包管理中的邀请奖励块。
4. 旧设置项是否后续并入新管理页，需要在新页面功能验证完成后再做收口判断。

## 核心业务规则

### 账号级返利方案锁定

这是本次设计中最重要的业务约束。

1. 用户第一次创建邀请链接前，必须先为当前账号选择返利方案。
2. 返利方案的选择对象是“账号”，不是“邀请链接”。
3. 账号一旦选定返利方案，该账号下所有邀请链接都继承同一方案。
4. 第一版中，账号已锁定的返利方案不允许修改。
5. 页面必须明确提示“方案一旦选择永久生效，无法更改”。

### 多邀请链接的业务语义

第一版中，多邀请链接的用途必须固定为“分发归因”，而不是“模式分流”。

1. 用户可以为不同分发平台创建多条邀请链接。
2. 多条邀请链接用于区分投放渠道，例如小红书、抖音、微信群、公众号、博客等。
3. 每条邀请链接后续需要支持单独查看邀请与返利统计。
4. 不同邀请链接之间不允许绑定不同返利模式。
5. 后续如果需要实现“不同链接支持不同模式”，应作为后续增强需求单独设计。

### 返利快照规则

管理员可以调整返利方案参数，但这类修改只能影响新产生的返利。

1. 已产生的佣金记录必须保存产生当时的方案快照。
2. 后续管理员修改方案参数时，历史佣金记录不得被反向污染。
3. 用户端展示的实际返利比例必须根据当前方案参数计算并展示。

## 用户页面设计

`邀请返利中心` 作为普通用户的独立业务页，建议采用多标签结构，统一承载返利相关能力。

### 总览区

页面顶部展示四项核心指标：

1. `总收益`
2. `可提现`
3. `待结算`
4. `邀请人数`

这四项指标用于帮助用户快速判断返利规模与当前可操作余额。

### 邀请链接标签页

该标签页承担两段式流程：首次选方案，以及后续链接管理。

#### 首次进入且账号未锁定方案

当用户账号尚未锁定返利方案时：

1. 页面优先展示返利方案选择卡片。
2. 用户必须先选择方案并确认。
3. 选择确认后，方案写入账号级锁定关系。
4. 在方案未锁定前，不允许直接创建邀请链接。

#### 账号已锁定方案后

当用户账号已经锁定方案时：

1. 页面展示“创建新邀请链接”表单。
2. 返利方案字段展示为只读状态。
3. 用户可创建多条邀请链接。
4. 多条邀请链接全部继承账号已锁定方案。

### 用户创建链接字段

第一版建议至少包含以下字段：

1. `链接用途/渠道备注`
2. `有效期`
3. `返利方案（只读展示）`

其中：

1. `链接用途/渠道备注` 用于记录该链接分发到哪个平台。
2. `有效期` 是否允许用户自定义，受管理员全局配置控制。
3. `返利方案` 跟随账号锁定关系，不允许用户在链接级修改。

### 邀请链接列表

第一版建议展示以下字段：

1. 邀请链接
2. 链接用途/渠道备注
3. 返利方案
4. 邀请人数
5. 成功人数
6. 返利金额
7. 状态
8. 有效期或过期时间

链接状态至少包括：

1. `生效中`
2. `已停用`
3. `已过期`

### 佣金记录标签页

用于查看返利结算明细，建议展示以下字段：

1. 被邀请用户
2. 层级
3. 充值金额
4. 返利额度
5. 返利方案
6. 状态

状态至少包括：

1. `待结算`
2. `已结算`
3. `已提现`

### 邀请用户标签页

用于查看被邀请用户清单，建议展示以下字段：

1. 用户名
2. 层级
3. 总充值
4. 总返利

### 提现管理标签页

该标签页包含提现申请表单与历史提现记录。

#### 提现申请表单

第一版建议包含：

1. 提现金额
2. 提现方式
3. 收款账号
4. 收款人姓名
5. 备注

#### 提现记录列表

第一版建议展示：

1. 提现金额
2. 手续费
3. 实际到账
4. 提现方式
5. 状态

状态至少包括：

1. `待审核`
2. `已批准`
3. `已完成`
4. `已拒绝`

## 管理员页面设计

`邀请返利管理` 作为管理员独立页面，建议采用三标签结构。

### 概览标签页

顶部展示平台级统计卡片，建议至少包含：

1. 活跃邀请人
2. 总邀请数
3. 待处理提现
4. 总返利支出

概览页下方建议放置：

1. 返利方案概览列表
2. 待审核提现列表

### 返利配置标签页

该标签页建议拆成两个配置区：

1. `全局规则`
2. `返利方案`

#### 全局规则配置项

第一版必须支持以下配置项：

1. `启用邀请返利系统`
2. `默认邀请链接有效期`
3. `是否允许用户自定义链接有效期`
4. `最低提现额度`
5. `提现手续费百分比`
6. `固定手续费`
7. `自动审批提现`
8. `用户月分佣上限开关`
9. `用户月分佣上限额度`

#### 返利方案配置项

第一版必须支持以下配置项：

1. `方案名称`
2. `方案类型`
3. `方案启用状态`
4. `方案说明`
5. `利润分配比例`
6. `最低利润率`
7. `一级返利占比`
8. `二级返利占比`

其中 `方案类型` 第一版保持与旧项目一致，至少支持：

1. `持续返利`
2. `一次性买断`

### 提现审核标签页

该标签页用于集中查看并处理提现申请，建议提供以下能力：

1. 查看提现申请列表
2. 查看申请用户、金额、手续费、到账金额、方式、账户、状态
3. 批准提现
4. 拒绝提现
5. 拒绝时填写原因

## 第一版明确固定，不开放为配置项的规则

以下规则在第一版中作为固定业务约束，不开放为管理员可调配置：

1. 不开放“方案锁定后允许修改”。
2. 不开放“链接级返利模式”。
3. 不开放“链接覆盖账号返利模式”。
4. 不开放“超出二级的返利层级”。
5. 不开放复杂自动审批策略与条件树。
6. 不开放链接级独立结算规则。
7. 历史记录默认长期保留，不做后台开关。

## 与当前项目现有能力的关系

### 可直接复用的现有能力

当前项目已有的以下能力可以作为基础复用：

1. 用户邀请相关基础字段与逻辑。
2. 钱包页中的邀请链接、复制链接、邀请额度划转能力。
3. 后台中的邀请奖励额度设置项。
4. 控制台侧边栏模块与入口控制机制。

### 需要新增或迁入的能力

本次需要新增或从旧项目思路迁入的能力包括：

1. 用户独立返利中心页面。
2. 管理员独立返利管理页面。
3. 账号级返利方案锁定关系。
4. 多链接分发归因与链接维度统计。
5. 佣金记录与状态流转。
6. 提现申请与管理员审核。
7. 返利方案配置与历史快照约束。

## 后续实现阶段的重点检查项

后续进入实现时，必须优先保证以下关键链路闭环：

1. 用户首次进入返利中心时的方案选择与账号锁定。
2. 账号锁定方案后创建多条邀请链接。
3. 多链接按用途区分，并能按链接查看邀请与返利统计。
4. 用户端佣金状态展示与提现申请提交流程。
5. 管理员端返利配置、方案编辑、提现审核闭环。
6. 钱包管理旧邀请奖励块在迁移期继续可用，不影响现有使用路径。

## 结论

本次设计的最终边界如下：

1. 新增两个独立页面：`邀请返利中心` 与 `邀请返利管理`。
2. 页面路由分别采用 `/console/referralcenter` 与 `/console/referralmanage`。
3. 用户返利模式按账号锁定，不按邀请链接锁定。
4. 多邀请链接用于不同平台分发和统计归因，不用于多模式并存。
5. 管理员第一版重点配置全局规则、返利方案与提现审核。
6. 钱包管理旧邀请奖励块先保留，待新页面稳定后再移除。
