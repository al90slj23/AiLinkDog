# gogogo memory 保存后直接结束设计

## 摘要

调整 `gogogo.memory.sh` 的记忆同步收尾交互：当用户在“总结与纯化计划预览”中选择“保存到知识库，并继续处理纯化”后，流程完成保存与工作日志更新后直接结束，不再进入“打开总结报告和所有纯化目标文件”的后续菜单。

## 目标

- 保存总结与纯化计划后直接结束当前 memory 同步流程。
- 不再默认要求用户选择是否打开总结报告或纯化目标文件。
- 避免在非交互终端或无图形编辑器场景下触发 Vim / 编辑器打开失败。

## 非目标

- 本次不改 DeepSeek 总结、预判、纯化计划生成逻辑。
- 本次不修改纯化目标文件识别逻辑。
- 本次不新增“按纯化计划自动执行纯化”的新流程。
- 本次不删除现有 `open_path_in_editor` 能力在其他位置的使用。

## 背景

当前 `run_memory_sync_flow()` 在成功保存总结后，会无条件调用 `run_purification_followup()`，弹出以下菜单：

```text
1. 现在打开总结报告和所有纯化目标文件
2. 只打开指定编号的纯化目标文件
3. 仅记录总结与纯化计划，不立即打开文件
0. 退出
```

这条交互默认把“打开文件”作为保存后的主路径，但当前实际使用方式已经变化：

- 用户先在终端里查看完整纯化计划；
- 确认后保存；
- 不希望再进入编辑器或文件打开流程；
- 更不希望在非终端编辑器环境中看到 `Vim: Input is not from a terminal` 之类错误。

因此，保存后的 follow-up 菜单已经不符合当前主流程，需要收敛。

## 方案对比

### 方案 A：保存后直接结束流程

做法：

- 成功保存后不再调用 `run_purification_followup()`；
- 仅输出已保存成功的提示，然后返回。

优点：

- 改动最小；
- 完全匹配当前“终端确认后结束”的使用方式；
- 彻底避开编辑器打开问题。

缺点：

- 当前这条入口下不再提供“顺手打开文件”的能力。

### 方案 B：保留菜单，但默认改成不打开文件

做法：

- 菜单继续保留；
- 将“不打开文件”改为默认或推荐选项。

优点：

- 保留旧能力。

缺点：

- 用户仍然要多走一步交互；
- 与当前“保存后就结束”的真实需求不完全一致。

## 推荐方案

采用方案 A。

## 设计细节

1. `run_memory_sync_flow()` 在成功写入：
   - 当前总结报告文件；
   - L4 工作日志文件；
   后直接返回。
2. 不再调用 `run_purification_followup()`。
3. `run_purification_followup()` 暂不删除，保留为后续可能重构或复用的旧能力，但本次主流程不再进入它。

## 测试设计

新增 shell 回归测试，验证：

1. 当 memory 流程完成保存后，不会再出现“现在打开总结报告和所有纯化目标文件”这类菜单文本。
2. 流程会在输出保存成功提示后直接结束。
3. 不会触发 `open_path_in_editor` 路径。

## 风险与边界

- 本次改动会让“保存后立即打开文件”的旧行为从主路径中消失；如果未来仍需要该能力，应以显式独立入口重新设计。
- 本次不尝试设计“保存后自动继续纯化”的新执行链路，避免把一个交互收敛任务扩展成新的自动化流程。
