# UpstreamTracking 产品化重构设计

## 背景

当前 `upstreamtracking` 已经具备一定的后端分析能力与证据链能力，包括：

- 上游 `new-api` commits 抓取
- compare/diff 摘要
- 远程 `.ai` 记忆抓取
- DeepSeek 分析
- 周期记录、事项、上下文落库

但页面当前仍然带有明显的“技术治理台”表达方式，核心问题不在于有没有数据，而在于页面与分析输出口径没有围绕真实维护决策场景设计。用户真正想看的不是 `provider`、`model`、`token`、`cycles` 这类底层字段，而是：

1. 当前从哪个 `new-api` 版本开始跟随
2. 最近一次跟随优化到了哪个版本
3. 本次上游更新主要改了什么
4. ALD 之前有没有做过类似优化
5. 这次是否建议合并
6. 如果建议合并，应直接修改哪些文件，还是需要适配式合并

因此，这次重构的目标不是继续在现有治理台页面上修补，而是把 `upstreamtracking` 重构为一个真正服务于维护决策的“上游更新评估页”。

## 目标

本次重构目标如下：

1. 保留现有分析能力与证据链能力
2. 重做页面专用输出结构，使前端不再直接消费零散技术字段
3. 将页面稳定收敛为三个标签页：`分析概览`、`分析历史`、`分析配置`
4. 将“是否建议合并”与“如何合并”升级为一级信息，而不是长文本附属说明
5. 让分析历史以“执行记录”为主线，而不是以固定周期窗口为主线

## 非目标

本次不做以下内容：

- 不推翻底层数据库主模型
- 不新增复杂任务调度系统
- 不做自动代码合并
- 不做复杂 diff 可视化阅读器
- 不做大规模本地代码扫描系统
- 不把历史主线改成周期窗口优先

本次重点是“输出口径与页面表达”重构，而不是基础设施扩建。

## 设计一：页面定位

`/console/upstreamtracking` 的定位由“技术治理台”调整为“上游更新评估页”。

页面应帮助维护者快速回答以下问题：

1. 当前 ALD 的上游跟随基线是什么
2. 最近一次跟随优化版本是什么
3. 本次上游更新做了什么
4. ALD 本地是否已有类似优化
5. 本次是否值得跟进
6. 如果跟进，应如何合并

页面依旧承载治理、分析和决策支持职责，但默认视角必须从“系统能力展示”切换为“维护者决策视角”。

## 设计二：页面信息架构

### 1. 分析概览

`分析概览` 只回答“这次值不值得看、值不值得跟”。

建议固定为三个区块：

#### A. 当前跟随基线

展示静态背景信息：

- 起始跟随的 `new-api` 版本
- 最近一次跟随优化版本
- 当前跟踪周期（天）
- 当前上游仓库与基线分支

#### B. 本次更新摘要

展示本次执行的核心结论：

- 本次执行时间
- 本次上游更新说明
- ALD 现有实现里是否已有类似优化
- 本次是否建议合并
- 风险等级或注意事项

#### C. 合并建议

展示合并落地方式：

- 建议直接修改的文件或模块
- 如果不能直接修改，明确说明需要适配式合并
- 合并方式建议：直接移植 / 局部重写 / 继续观察 / 暂不跟进

本标签页不再直接展示 `provider`、`model`、`token`、`cycles` 等技术字段作为主信息。

### 2. 分析历史

`分析历史` 只回答“以前做过什么判断”。

建议固定为两个主区块：

#### A. 执行记录列表

按执行记录展示，每条记录保留这些列：

- 执行时间
- 本次更新一句话标题
- 是否建议合并
- 合并方式建议
- 当前决策状态

历史主线采用“按执行记录”，不以固定周期窗口作为页面主轴。

#### B. 执行记录详情

点开某条记录后展示：

- 本次更新说明
- ALD 是否已有类似优化
- 是否建议合并
- 为什么建议或不建议
- 建议改哪些文件或区域
- 决策记录
- 证据链入口

`待同步事项` 与 `分析上下文` 从属于当前执行记录，不再抢占页面主视角。

### 3. 分析配置

`分析配置` 只回答“系统按什么口径在跑”。

建议保留两段结构：

#### A. 当前使用中配置

直接展示当前生效值。

#### B. 编辑配置

支持编辑，并在输入框中用灰色占位显示当前值。

字段顺序调整为：

##### 业务配置

- 起始跟随版本
- 最近一次跟随优化版本
- 跟踪周期（天）

##### 高级分析配置

- 启用上游跟踪
- 仓库 Owner
- 仓库名称
- 基线分支
- 分析提供商
- 分析模型
- 分析 Base URL
- 分析 Token
- 分析范围

## 设计三：后端页面专用输出结构

不建议继续让前端直接消费底层 `cycle / action / context` 的零散字段。保留底层模型，但新增页面专用聚合结果（ViewModel）。

### 每条执行记录的核心背景字段

- `title`
- `executedAt`
- `startVersion`
- `lastSyncedVersion`
- `intervalDays`

### 本次更新结论字段

- `updateSummary`
- `hasSimilarLocalWork`
- `localWorkSummary`
- `shouldMerge`
- `mergeReason`
- `riskSummary`

其中：

- `shouldMerge` 建议使用稳定枚举，例如：`merge` / `observe` / `skip`
- `hasSimilarLocalWork` 应作为显式字段存在，而不是隐藏在长文本中

### 合并方式建议字段

- `mergeStrategy`
- `targetFiles`
- `targetAreas`
- `mergePlanSummary`

建议 `mergeStrategy` 使用稳定枚举：

- `direct_patch`
- `adaptive_merge`
- `rewrite_reference`
- `observe_only`

### 证据链继续保留，但降级为从属信息

继续保留：

- upstream commits
- compare/diff
- `.ai` 远程记忆
- analysis prompt
- analysis response
- actions

但它们不再作为页面主输出，而是作为详情展开和审阅依据。

### 聚合返回建议

建议后端返回面向页面的聚合结构，例如：

- `overview`
- `historyItems`
- `configView`
- `evidence`

前端只渲染该聚合结构，不再自己拼底层字段。

## 设计四：分析链路收敛方式

建议将分析链路拆成四个阶段。

### 1. 上游变化整理

输入：

- 最近一段时间的 commits
- compare/diff 摘要
- 文件级变化列表

输出：

- `upstreamChangeSummary`
- `changedFiles`
- `changedAreas`

目标是先把“上游做了什么”整理成稳定输入，而不是直接把原始提交暴露给页面。

### 2. ALD 本地对照整理

输入：

- `.ai` 远程记忆关键文件
- 与本次主题相关的 ALD 规则与知识
- 必要时补少量代码区域摘要

输出：

- `localKnowledgeSummary`
- `possibleExistingWork`
- `possibleTouchedAreas`

目标是尽量让服务层先做相关上下文收敛，而不是把海量原文直接扔给模型。

### 3. 评估推理

模型必须明确回答以下问题：

1. 本次 upstream 更新核心在改什么
2. ALD 现有实现里有没有类似优化或冲突点
3. 这次是否建议跟进
4. 如果跟进，应该直接改哪些文件或模块
5. 如果不能直接改，建议采用什么合并方式

Prompt 必须强制输出结构化字段，而不是泛泛而谈。

### 4. 结果归一化

服务层负责：

- 补默认值
- 校正枚举值
- 统一数组结构
- 拆分长文本为稳定字段
- 保证页面可直接渲染

前端不应承担模型输出纠偏职责。

## Prompt 约束

Prompt 中应显式约束模型：

- 不要只复述 commit message
- 不要给空泛总结
- 不要建议直接整体拷贝上游代码
- 必须判断 ALD 是否已有类似优化
- 必须给出明确合并态度
- 如果建议合并，必须指出目标文件或目标模块
- 如果结构差异大，必须明确写“适配式合并”或“参考重写”

## 失败与降级策略

### 1. compare 获取失败

- 允许继续分析
- 记录“本次分析缺少 compare 证据”

### 2. 远程 `.ai` 记忆获取失败

- 优先回退本地 `.ai`
- 若仍失败，允许继续分析，但标记“未纳入完整 ALD 记忆上下文”

### 3. 模型输出缺字段

- 服务层补默认值
- 页面不因为字段缺失而崩溃

### 4. 分析失败

- 历史记录中保留失败执行记录
- 不把失败当成“从未执行”

## 设计五：实施边界与验收标准

### 本次第一阶段要做的事

1. 重做页面专用 ViewModel
2. 重做分析输出结构
3. 重做前端三个标签页
4. 保留证据链，但降级为从属信息

### 本次第一阶段不做的事

- 不重构底层数据库主模型
- 不新增复杂调度系统
- 不做自动合并代码
- 不做复杂 diff 阅读器
- 不把历史主线改成周期窗口优先

### 验收标准

#### 分析概览

页面应直接显示：

- 当前跟随基线
- 最近一次跟随优化版本
- 本次更新说明
- 是否建议合并
- 合并方式建议
- 建议改哪些文件或模块

#### 分析历史

页面应支持按执行记录查看：

- 执行时间
- 本次更新标题/摘要
- 是否建议合并
- 合并方式
- 决策状态
- 详情与证据链入口

#### 分析配置

页面应清晰显示：

- 当前使用中配置
- 编辑配置
- 起始跟随版本 / 最近一次跟随优化版本 / 跟踪周期（天）优先展示

#### 分析结果输出

执行一次分析后，后端应稳定产出：

- `updateSummary`
- `hasSimilarLocalWork`
- `localWorkSummary`
- `shouldMerge`
- `mergeStrategy`
- `targetFiles`
- `targetAreas`
- `mergeReason`
- `riskSummary`

### 建议测试

1. 服务层测试：验证新 ViewModel 字段与归一化逻辑
2. 控制器测试：验证页面聚合结果结构稳定
3. 前端验证：确认三个标签页按新业务语义展示

## 建议实施顺序

1. 后端定义新的页面输出结构
2. 调整分析结果归一化
3. 新增页面聚合接口或聚合返回
4. 前端重做三个标签页
5. 清理页面对旧技术字段的直接依赖

## 结论

本次重构不是删除现有分析能力，而是保留现有能力与证据链，在其上新增一层产品化结果结构，把 `upstreamtracking` 从“技术治理台”重构为“上游更新评估页”。
