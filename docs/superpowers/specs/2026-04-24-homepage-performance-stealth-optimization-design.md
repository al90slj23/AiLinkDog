# 2026-04-24 Homepage Performance Stealth Optimization Design

## 背景

当前首页的主要性能与耗能热点集中在桌面端首屏 Hero 区域，尤其是机器人 3D 场景、全局鼠标跟踪和持续特效叠加后的 CPU/GPU 活跃度。现有实现已经具备两项正确的基础策略：

- 3D 机器人仅在桌面端启用
- 3D 机器人仅在进入视口后再懒加载

但在进入可见状态后，首页仍然存在以下持续成本：

- `SplineScene` 在窗口级别监听 `pointermove`，并把事件桥接到 3D 画布
- `Spotlight` 在窗口级别监听 `mousemove`
- 首页滚动持续更新 CSS 自定义变量
- 多个视觉特效在前台和后台场景下没有区分对待

用户要求是严格无感优化：正常桌面使用、滚动、鼠标移动、停留观看时，用户层面不应感知到交互或视觉效果差异；只允许在后台标签页、非交互区域、无意义更新路径上降低成本。

## 目标

本次优化目标如下：

1. 不改变首页视觉风格、布局结构、文案内容、3D 机器人存在与呈现方式。
2. 不改变正常前台桌面交互时的主观观感，不引入可察觉的延迟、卡顿或效果退化。
3. 降低首屏 Hero 在桌面端的无效事件处理、无意义样式写入和后台能耗。
4. 保持现有懒加载策略，避免把优化扩大为视觉重构或 3D 资源重做。

## 非目标

以下内容不在本次方案范围内：

- 替换 Spline 技术栈
- 删除 3D 机器人或改成纯静态图
- 修改首页视觉设计、毛玻璃层级、文案和模块排布
- 对全站通用动画系统做统一重构
- 以牺牲桌面前台交互体验为代价换取更强节能

## 方案对比

### 方案 A：调度层无感优化

仅修改交互监听范围、桥接频率、页面可见性处理和滚动写入策略，不改动最终视觉结果。

优点：

- 最符合“严格无感”要求
- 改动集中，回归面较小
- 可以直接命中当前最明显的无效开销点

缺点：

- 无法从根本上消除 3D 场景本身的 GPU 成本
- 节能收益取决于用户是否大量停留在首页首屏

### 方案 B：分级交互模式

为机器人场景增加低交互态和高交互态，在用户接近交互时再切换到高交互模式。

优点：

- 理论节能收益更高
- 更适合长时间停留首页但不与机器人交互的场景

缺点：

- 极少数敏感用户可能感受到激活延迟或跟手差异
- 不满足当前“严格无感”的约束

### 方案 C：削减视觉合成成本

通过减少 blur、backdrop-filter、外围动画等方式降低 GPU 压力。

优点：

- 实现直接
- 对低性能设备有一定帮助

缺点：

- 更容易引入视觉细节变化
- 对当前最大热点的命中不如方案 A 直接

## 选型

本次采用方案 A，并吸收方案 C 中完全无感的后台暂停策略，不采用方案 B。

最终策略是：保留当前首页前台视觉与交互结果，专注消除“用户并不需要但系统仍在持续做”的工作。

## 详细设计

### 1. SplineScene 输入桥接只在机器人交互区域内激活

当前 `SplineScene` 在窗口级别监听 `pointermove` 与 `mouseenter`。这意味着用户在首页其他区域移动鼠标时，也会触发机器人画布的事件桥接。

优化后：

- 鼠标进入机器人区域后才启用输入桥接
- 鼠标离开机器人区域后停止桥接
- 桥接目标仍然是同一个 3D `canvas`
- 机器人在交互区域内的跟手视觉保持不变

这样可以把事件处理范围从“整个窗口”收缩到“用户真正与机器人交互的区域”。

### 2. SplineScene 鼠标桥接增加高流畅上限

当前鼠标桥接通过 `requestAnimationFrame` 合并事件，但理论上仍可能在每个动画帧都推送一次桥接动作。

优化后：

- 保留动画帧合并思路
- 在桥接层增加一个高流畅上限，避免高频鼠标输入下的过度调度
- 该上限应选择在人眼和普通用户操作中难以察觉差异的范围内

这个上限仅作用于“外部输入桥接频率”，不直接修改 Spline 内部渲染逻辑，也不改变机器人本身的视觉素材和动作资源。

### 3. SplineScene 增加页面可见性节流

优化后：

- 页面不可见时暂停输入桥接
- 回到前台时自动恢复
- 不尝试在后台维持无意义交互同步

这部分变化对用户完全不可见，但能够显著减少后台标签页状态下的无效唤醒和渲染驱动。

### 4. Spotlight 改为容器级监听

当前 `Spotlight` 使用窗口级 `mousemove` 更新聚光位置。实际上首页只需要在其所属 Hero 容器内响应该效果。

优化后：

- 只监听 Hero 容器的指针移动
- 指针离开容器时隐藏 Spotlight
- Spotlight 的尺寸、透明度、位置动画方式保持不变

这样既保留原有效果，也减少页面其他区域鼠标移动带来的无效更新。

### 5. 滚动驱动 CSS 变量写入去除无效更新

当前首页滚动时会更新 `--ald-scroll-progress` 和 `--ald-scroll-y`。现有逻辑已经通过 `requestAnimationFrame` 合并了 `scroll`，这是正确基础，但仍可进一步减少无意义写入。

优化后：

- 仅当计算值相较上次达到最小变化阈值时，才写入 CSS 变量
- 保持现有滚动视觉节奏不变
- 不修改现有 reveal 和进度条机制

该优化本质上是去掉“值虽重新计算但用户根本不可能看出差异”的样式写入。

### 6. 后台场景统一暂停非必要交互驱动

对于仅服务于前台观感的交互驱动，本次统一增加页面可见性感知：

- 页面隐藏时暂停非必要输入驱动
- 页面恢复可见时无缝恢复
- 不对前台用户路径做行为降级

这条规则主要作用于首页首屏 Hero 内的交互型效果，而不是全站任意动画系统。

## 组件边界与改动范围

预计改动集中在以下位置：

- `web/src/components/ui/spline-scene.jsx`
- `web/src/components/ui/spotlight.jsx`
- `web/src/pages/Home/components/DefaultLandingPage.jsx`
- 与上述逻辑直接相关的测试文件

本次不应扩散到首页其他信息模块或无关页面，避免把性能优化演变成结构性重构。

## 数据流与状态约束

### SplineScene

- 输入来源：机器人可交互容器内的指针事件
- 条件门禁：桌面端、场景已加载、页面可见、指针位于交互区域
- 输出：桥接后的 `pointermove` / `mousemove` 到内部 `canvas`

### Spotlight

- 输入来源：Hero 容器内的指针位置
- 条件门禁：容器存在、页面可见、指针在容器范围内
- 输出：聚光位置样式更新

### Scroll Metrics

- 输入来源：页面滚动
- 条件门禁：值变化超过阈值
- 输出：`--ald-scroll-progress` 与 `--ald-scroll-y`

## 错误处理

- 若 `Spline` 画布尚未准备好，不应报错或重复调度，直接跳过当前桥接周期。
- 若页面可见性切换频繁，恢复逻辑必须幂等，避免重复注册事件。
- 若 Spotlight 父容器不可用，应安全降级为不显示该效果，而不是绑定到错误节点。
- 若滚动变量阈值逻辑失效，最坏情况应退化为现有行为，而不是导致首页样式不更新。

## 测试与验证

至少覆盖以下验证：

1. 首页桌面端首屏仍能正常渲染机器人区域。
2. 机器人在交互区域内仍可保持原有鼠标跟随机制，不出现明显迟滞。
3. 鼠标在页面其他区域移动时，不再触发机器人相关桥接。
4. Spotlight 仅在其容器内激活，离开容器后隐藏。
5. 页面进入后台后，非必要交互驱动暂停；回到前台后恢复。
6. 首页现有结构和关键 class 不变，避免破坏已有布局测试。
7. 前端构建通过。

## 风险与回归点

- 如果机器人区域边界判断过于严格，可能出现边缘区域跟手不稳定。
- 如果桥接节流配置过强，少数敏感用户可能感觉到交互变钝。
- 如果可见性恢复处理不严谨，切回前台后可能出现一次性失焦或未恢复。
- 如果 Spotlight 改为容器监听时未正确处理定位上下文，可能导致光斑坐标偏移。

因此实现时应优先选择保守阈值和局部最小改动，确保“用户无感”优先于“理论最省”。

## 实施原则

1. 先收缩监听范围，再做频率限制，不一开始同时大改两层逻辑。
2. 保持前台桌面观感等价，把后台和无效交互路径作为主要节能入口。
3. 仅在现有组件内局部修正，不新增复杂抽象或跨页面性能框架。
4. 每一项优化都要能回答：如果去掉它，用户能否明显看出区别；如果答案是能，就不属于本次范围。
