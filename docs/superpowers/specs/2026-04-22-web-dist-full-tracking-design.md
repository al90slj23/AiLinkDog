# web/dist 完整跟踪策略设计

## 摘要

将当前仓库的 `web/dist` Git 跟踪策略从“半跟踪”调整为“完整跟踪”，以消除 `web/dist/index.html` 与哈希 `assets/*` 之间的策略不一致问题。

本次只调整版本管理策略，不修改前端源码，不改动后端 `embed` 实现，不改动前端构建产物内容本身。

## 背景

当前仓库中：

- Go 后端通过 `//go:embed web/dist` 直接依赖 `web/dist`
- `web/dist` 是前端构建产物，也是后端编译输入

但根 `.gitignore` 当前采用的是“半跟踪”策略：

```gitignore
web/dist/
!web/dist/
!web/dist/index.html
```

这意味着：

- `web/dist/index.html` 被跟踪；
- 哈希资源文件 `web/dist/assets/*` 处于不稳定、非完整跟踪状态；
- 每次构建后会产生大量删除/新增噪音。

## 目标

- 让 `web/dist` 的 Git 策略与当前后端 embed 结构保持一致。
- 消除“只跟踪 index.html、不完整跟踪 assets”的长期噪音来源。
- 保持当前仓库“源码拉下即可具备可 embed 前端产物”的使用方式。

## 非目标

- 本次不把 `web/dist` 改成完全忽略。
- 本次不重构 `gogogo`、CI、后端构建流程。
- 本次不试图减少 `dist` 产物文件数量。

## 方案对比

### 方案 A：完整跟踪 `web/dist`

做法：

- 去掉当前对 `web/dist/index.html` 的特殊半跟踪规则；
- 调整为允许完整跟踪 `web/dist`；
- 将当前 `web/dist/assets/*` 的新增/删除作为正式构建产物变更纳入版本管理。

优点：

- 与后端 `embed` 结构一致；
- Git 策略自洽；
- 不要求引入“先构建前端再构建后端”的新前置条件。

缺点：

- 仓库会持续包含构建产物；
- 前端构建变更会带来较大 diff。

### 方案 B：彻底不跟踪 `web/dist`

做法：

- 完整忽略 `web/dist`；
- 所有后端构建前必须先生成前端产物。

优点：

- 仓库最干净。

缺点：

- 需要同步重构构建/运行规则；
- 不适合作为当前最小调整。

## 推荐方案

采用方案 A：完整跟踪 `web/dist`。

## 设计细节

1. 调整根 `.gitignore` 中 `web/dist` 相关规则，去掉当前“半跟踪”模式。
2. 保持 `web/dist` 及其 `assets/*` 作为完整构建产物一起纳入版本管理。
3. 后续若 `web/dist` 需要更新，应把整套变更视为一次前端构建产物更新，而不是只看 `index.html`。

## 风险与边界

- 本次只是让 Git 跟踪策略自洽，不代表之后每次都必须主动提交 `web/dist`；是否提交仍取决于具体任务。
- 如果未来决定切换到“彻底不跟踪 `web/dist`”，那会是一次单独的构建策略改造任务。
