# L3 standards 文档命名整改设计

## 背景

当前 `.ai/L3#Standards/standards/` 下的正文文件已经具备初步分组意识，例如：

- `00.meta-01.core.md`
- `02.backend-01.json-and-dto.md`
- `02.backend-03.relay-and-channel.md`
- `10.ai-memory-02.usage.md`

但现状仍存在几个问题：

1. 命名结构尚未完全统一，部分层级使用复合词，部分主题使用并列短语。
2. 文件名经常直接暴露“and”式复合主题，说明单篇文档承载了多个主主题。
3. 当前命名方式对未来持续细分的支持不够稳定，文档一旦继续拆分，容易出现层级与命名风格混杂。
4. 文件名中的英文标识未被严格约束为“阅读正文后提炼出的精准主题词”，后续容易退化为凭旧文件名或主观理解命名。

由于 `L3#Standards` 是长期规范层，而正文文件又受到单文件行数限制，因此文档未来会持续拆细。本次需要建立一套能够长期扩展的命名规则，并将其作为后续整改与新增文档的统一基线。

## 目标

1. 统一 `.ai/L3#Standards/standards/` 正文文件的命名结构。
2. 让命名天然支持四层、五层或更深层级的持续细分。
3. 让每一层命名都能表达稳定语义路径，而不是依赖单层复合短语兜底。
4. 要求英文标识必须在阅读全文后归纳得出，避免拍脑袋命名。
5. 通过命名规则倒逼正文文件维持单主题结构，减少混合型规范文档。

## 非目标

1. 本次不强制将 `README.md` 纳入正文命名规则。
2. 本次设计不要求同步重构 `L3#Standards` 之外的其他层级目录。
3. 本次设计不将目录树改造成多层目录结构，整改对象仍以 `standards/` 平铺正文文件为主。
4. 本次设计不以“全局唯一编号”作为目标，编号只服务于父层内排序。

## 适用范围

本规则仅适用于：

- `.ai/L3#Standards/standards/` 下的规范正文文件。

不强制适用于：

- `.ai/L3#Standards/README.md`
- `.ai/L3#Standards/standards/README.md`
- 其他 `.ai/` 层级目录

`README.md` 类文件继续作为导航与说明文件保留，不纳入该正文命名强规则。

## 推荐方案

采用“可无限扩展的单词链命名”。

正式格式：

`序号.单词-序号.单词-序号.单词.md`

例如：

- `00.meta-01.core.md`
- `01.arch-02.layers.md`
- `02.backend-01.json.md`
- `02.backend-01.json-01.wrapper.md`
- `02.backend-03.relay-01.channel-01.stream.md`
- `10.ai-01.memory-01.usage.md`

该方案不是固定三层，而是允许在需要时继续增加层级，例如：

- `02.backend-03.relay-01.channel-01.stream-01.options.md`

## 命名结构模型

### 结构原则

1. 正文文件统一使用链式命名，而不是短语式命名。
2. 每一层都采用 `序号.单词` 的固定结构。
3. 层与层之间统一使用 `-` 连接。
4. 层级不设上限，可按文档拆分需要持续向下扩展。
5. 文件整体语义由整条链共同表达，不由最后一个词单独承担。

### 单层约束

1. 每一层默认只放一个英文单词。
2. 不使用层内连字符将多个英文单词拼成复合短语。
3. 不将 `stream-options`、`platform-files`、`rule-sources` 这类复合词作为常规层级名称。
4. 如果一个层级需要两个词才能表达清楚，优先考虑继续增加一层，而不是在单层中堆词。

因此，推荐写法是：

- `02.backend-03.relay-01.channel-01.stream-01.options.md`

而不是：

- `02.backend-03.relay-01.channel-01.stream-options.md`

### 语义表达原则

1. 单个词不要求脱离上下文后仍能独立完整自解释。
2. 只要放入前置层级链后语义已经足够精准，该单词就可以成立。
3. 精准优先，不追求单层名称看起来像一个完整短语。

例如：

- `10.ai-01.memory-01.usage.md` 中的 `usage` 单独看较宽，但放在 `ai -> memory` 链中已足够精准。

## 单词提炼规则

命名时，每一层中的单词都必须来自对正文内容的阅读与归纳，而不是直接继承旧文件名。

### 基本规则

1. 必须先阅读正文，再提炼命名单词。
2. 单词必须概括该层级下的真实主主题。
3. 单词以“精准”优先，不以“完整短语感”优先。
4. 允许使用短词，但必须在前置层级链中语义清晰。
5. 不允许仅根据旧文件名、目录位置或主观猜测直接命名。
6. 不允许把旧文件名中的复合短语机械拆开后直接塞入层级。

### 命名判断顺序

对任意一篇正文文件，命名时应按以下顺序判断：

1. 这篇文档主要在规定什么？
2. 它属于哪条父层级链？
3. 在当前父层下，用哪个单词最精准？
4. 如果没有任何单个词足够精准，是否应该继续增加一层？
5. 如果增加层级仍无法表达，是否说明正文本来就应该拆成多篇？

### 合格标准

判断一个命名是否成立，不是看：

- 最后一个词能否脱离上下文单独完整表达主题

而是看：

- 整条链连起来，是否已精准表达正文主主题

## 拆文判定规则

本次整改不应只是批量改名，而应先判断正文是否已经形成单主题结构。

### 应优先拆文的情况

1. 文件标题或正文主结构明显包含并列主题。
2. 文件名或正文中出现稳定的 `and` 式复合主题。
3. 正文同时承载两个或多个相对独立的规则对象。
4. 阅读正文后无法为当前层提炼出一个足够精准的单词。
5. 后续明显还会继续细分，但当前文档已经把多个层级揉在一起。

例如，下列现有文件名都属于高风险整改对象：

- `02.backend-01.json-and-dto.md`
- `02.backend-03.relay-and-channel.md`
- `03.frontend-02.i18n-and-web-build.md`
- `09.tool-02.rule-sources-and-platform-files.md`

### 不一定要拆文的情况

1. 正文始终围绕一个核心对象，只是内部含多个子章节。
2. 次要概念仅用于辅助解释主主题，不构成独立长期规则对象。
3. 当前内容虽可继续细分，但暂时不足以形成独立维护单元。

### 拆文优先级

当出现以下二选一时，应优先选择“拆文或增层”：

- 用一个更长、更复杂的名字去兜住多个主题
- 先把内容拆清，再按单词链命名

本设计明确选择后者。

## 编号分配与排序原则

### 基本原则

1. 编号只在直属父层内排序，不做全局连续编号。
2. 编号表达的是同层顺序，不承担文档全局唯一 ID 的职责。
3. 同层排序优先体现阅读顺序与认知顺序，而不是纯字母顺序。

例如：

- `02.backend-01.json.md`
- `02.backend-02.database.md`
- `02.backend-03.relay.md`

在 `backend` 下重新编号；而在 `relay` 下又重新编号：

- `02.backend-03.relay-01.channel.md`
- `02.backend-03.relay-02.provider.md`

### 推荐排序逻辑

同一父层下，优先按以下顺序考虑编号：

1. 总则或核心底座
2. 基础机制
3. 关键子系统
4. 扩展、工具、协作类内容

更具体地说，排序应尽量符合：

- 先基础，再进阶
- 先总则，再专题
- 先依赖少的，再依赖多的
- 先稳定底座，再延伸细则

### 稳定性原则

1. 不因局部措辞优化而轻易改号。
2. 新增同层文档时，优先放在合适位置，但避免频繁整体洗牌。
3. 编号目标是长期可维护，而不是一次性形式最整齐。

## 存量文档整改流程

### 总原则

整改单位不是“旧文件名”，而是“旧文件中的规则内容”。

因此，整改顺序必须是：

`阅读 -> 判断是否拆分 -> 确定层级 -> 提炼单词 -> 分配编号 -> 重命名 -> 修复引用`

### 单篇文档整改步骤

1. 阅读正文，理解该文档真正规定了什么。
2. 提炼主主题，判断是否存在唯一主主题。
3. 判断是否需要拆文。
4. 确定该文档所属的完整语义链。
5. 为每一层提炼单词。
6. 在直属父层中分配编号。
7. 重命名文件。
8. 修复 README、索引与交叉引用。

### 推荐整改顺序

为减少反复改号，建议按“从粗到细”的顺序推进：

1. 先确定顶层大类顺序。
2. 再确定每个顶层的直属子层。
3. 最后处理更深层级的细分主题。

也就是说，应先稳定：

- `meta`
- `arch`
- `backend`
- `frontend`
- `quality`
- `tool`
- `ai`

再继续处理各分支下的细层级。

## 对现有文件的整改含义

本规则落地后，现有文件中所有“and 式”命名都不应被简单翻译为新的单词链，而应被视为：

- 需要阅读全文
- 需要确认是否拆分
- 需要重建层级语义

例如：

- `02.backend-01.json-and-dto.md`
  - 可能拆为 `02.backend-01.json.md`
  - 或拆为 `02.backend-01.json-01.wrapper.md` 与 `02.backend-02.dto.md`

- `02.backend-03.relay-and-channel.md`
  - 应先判断 `channel` 是 `relay` 下的子层，还是并列主题
  - 若是子层，应改造为 `02.backend-03.relay-01.channel.md`

- `03.frontend-02.i18n-and-web-build.md`
  - 大概率应拆为两个单主题文档，而不是继续保留并列关系

## 迁移注意事项

1. 不应在未阅读全文前直接分配新名字。
2. 不应将复合词仅拆成多个连字符片段后继续维持同一层级。
3. 不应为了追求“名字短”而牺牲语义精准。
4. 不应为了追求“结构整齐”而过度拆分尚未形成独立主题的内容。
5. 完成整改后，需要统一检查 README、索引、链接与交叉引用是否仍然正确。

## 最终结论

`.ai/L3#Standards/standards/` 正文文件应统一采用“可无限扩展的单词链命名”：

`序号.单词-序号.单词-序号.单词.md`

其中：

1. 每层只使用一个英文单词。
2. 单词必须在阅读全文后归纳得出。
3. 语义精准优先于短语完整感。
4. 文件整体语义由整条链共同表达。
5. 无法自然命名时，应优先考虑增层或拆文，而不是继续使用复合短语。

这套规则的本质不是“给旧文件换一种写法”，而是为 `L3 standards` 建立一套可以长期演化、持续细分、并能反向约束文档结构质量的命名基线。
